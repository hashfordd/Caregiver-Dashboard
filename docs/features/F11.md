# F11 — Per-Patient Alert Rules

## Goal

A caregiver opens the **Alert rules** tab on a patient's dashboard, configures rules across the four V1 rule types, and watches a "would have alerted in last 24 h" preview update as values change. Rule edits propagate to the live engine within 30 seconds. A saved rule fires within its expected latency on the next matching data point and surfaces in [F12](./F12.md)'s feed.

## Acceptance criteria

- The settings tab exposes **4 rule types in V1**: `zone`, `vitals`, `fall`, `inactivity`. The `repetitive_movement` enum value already exists in the migration and is **deferred to V2** — see [BACKLOG.md](../../BACKLOG.md). Do not pad the UI to imply 5 are coming. Spec note: PHASES.md Phase 4 originally read "5 distinct rule types"; that line is restated here in line with the deferral.
- Each rule type has a dedicated card: severity selector (`info` / `warn` / `critical`), enable toggle, type-specific parameter form, optional cooldown override, and a destructive delete control.
- The **live preview** runs the same evaluator against the last 24 h of stored data and shows a count + per-severity breakdown of the would-have-fired events. The preview re-runs on every parameter change without writing to the `alerts` table.
- A rule edit becomes effective within 30 s. Either the live engine reads `alert_rules` per evaluation (no cache) or its cache TTL is ≤ 30 s.
- Cooldown defaults follow [CROSS_CUTTING.md §3](../CROSS_CUTTING.md#3-alert-cooldowns); per-rule overrides are stored in `alert_rules.params.cooldown_seconds`.
- The acknowledge flow itself is owned by [F12](./F12.md); F11 is concerned only with rule configuration and preview.

## Rule type taxonomy

The V1 rule types, their trigger surfaces, source data, statefulness, and notes for the evaluator. Aligned with [PHASES.md Phase 4](../PHASES.md#phase-4--alert).

| Type                  | Trigger source              | Source data                                                              | Stateful?                                      | Evaluator notes                                                                                                                                                                                                                      |
| --------------------- | --------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `vitals`              | DB webhook on INSERT        | `sensor_readings`                                                        | stateless                                      | Each rule names a metric (`hr_bpm`, `spo2_pct`, `temp_c`) and a numeric range. Fire when the incoming value falls outside the range.                                                                                                 |
| `zone`                | DB webhook on INSERT        | `position_estimates` (canvas mode) + the rule's geofence polygon         | stateless except dwell-time                    | Point-in-polygon on the new position. If `dwell_seconds > 0`, require the patient inside (or outside, depending on `direction`) for that long — read the last `dwell_seconds` of history to confirm.                                 |
| `fall`                | DB webhook on INSERT        | `events` (where `type = 'fall'`)                                         | stateless                                      | Fire on every fall event for the patient unless suppressed by cooldown. Severity defaults to `critical`.                                                                                                                             |
| `inactivity`          | scheduled cron (per minute) | `position_estimates` — looks for the **absence** of motion over a window | stateful — needs the absence of recent updates | Run by the `inactivity_scan` function. For each enabled rule, compute the time since the last `position_estimates` row whose canvas delta from the previous row exceeds the rule's motion floor. Fire if delta > `inactive_minutes`. |
| `repetitive_movement` | (V2 deferred)               | —                                                                        | —                                              | Enum value exists in the migration. UI shows no card for this in V1. Tracked in [BACKLOG.md](../../BACKLOG.md).                                                                                                                      |

### Inactivity is the awkward one

Inactivity fires on the absence of inserts, so a DB webhook can't trigger it. Two implementation options sit on the table:

1. **Scheduled Edge Function via `pg_cron`** — every 60 s, a single function loops over enabled inactivity rules and fires per the threshold check. Cleanest. One pg_cron job, regardless of how many rules exist.
2. **Compute-on-write motion flag** — add a derived "moved" boolean to `position_estimates` and have a watchdog scan for patients with no recent `moved = true` rows. Doubles the write surface area for marginal benefit.

**Decision: Option 1.** A `pg_cron` job runs `inactivity_scan` every minute. The function loops over `alert_rules where type = 'inactivity' and enabled = true`, evaluates each, applies cooldown, writes to `alerts`. Lives at `apps/edge/functions/inactivity_scan/`.

**Fallback if `pg_cron` is not on the chosen Supabase plan**: schedule the same Edge Function via an external scheduler — Vercel cron, GitHub Actions on a 1-minute cron, or Supabase scheduled triggers (if released). The function code does not change. Document the chosen path in the migration's PR description, and capture the deployment env var(s) for the external scheduler in `.env.example`. Confirm pg_cron availability on the project's Supabase plan **before** locking the migration.

## Contracts in `packages/shared`

F11 owns the `AlertRule` discriminated union. The `type` field is the discriminator; each variant carries a typed `params` object. The shared `id`, `patient_id`, `severity`, `enabled`, `created_at`, `updated_at` fields mirror the `alert_rules` columns.

```ts
import type { Severity } from './severity.js';

export type AlertSeverity = 'info' | 'warn' | 'critical';

export interface AlertRuleBase {
  id: string;
  patient_id: string;
  severity: AlertSeverity;
  enabled: boolean;
  created_at: string;
  updated_at: string;
}

export interface ZoneRule extends AlertRuleBase {
  type: 'zone';
  params: {
    /** Polygon in floor-plan canvas coordinates: [[x, y], ...]. Closed implicitly. */
    polygon: [number, number][];
    /** 'enter' fires on entering the polygon; 'exit' fires on leaving. */
    direction: 'enter' | 'exit';
    /** Seconds the condition must hold continuously before firing. 0 = immediate. */
    dwell_seconds: number;
    /** Optional override of the severity-default cooldown. */
    cooldown_seconds?: number;
  };
}

export interface VitalsRule extends AlertRuleBase {
  type: 'vitals';
  params: {
    metric: 'hr_bpm' | 'spo2_pct' | 'temp_c';
    /** Inclusive bounds. Either may be null to make a one-sided range. */
    min: number | null;
    max: number | null;
    cooldown_seconds?: number;
  };
}

export interface FallRule extends AlertRuleBase {
  type: 'fall';
  params: {
    cooldown_seconds?: number;
  };
}

export interface InactivityRule extends AlertRuleBase {
  type: 'inactivity';
  params: {
    inactive_minutes: number;
    /** Optional time-of-day window in HH:mm-HH:mm caregiver-local. */
    only_between?: { from: string; to: string };
    /** Canvas-distance floor below which a position change does not count as motion. */
    motion_floor_px?: number;
    cooldown_seconds?: number;
  };
}

export type AlertRule = ZoneRule | VitalsRule | FallRule | InactivityRule;

export type DataPoint =
  | { kind: 'sensor_reading'; row: SensorReadingRow }
  | { kind: 'position_estimate'; row: PositionEstimateRow }
  | { kind: 'event'; row: EventRow }
  | { kind: 'tick'; at: string }; // used by inactivity_scan

export interface HistoryWindow {
  /** Recent rows the evaluator may need (zone dwell, inactivity). Caller-loaded. */
  positions: PositionEstimateRow[];
  sensors: SensorReadingRow[];
  events: EventRow[];
}

export type EvaluatorResult =
  | { fire: true; severity: AlertSeverity; context: Record<string, unknown> }
  | { fire: false };

export function evaluateRule(
  rule: AlertRule,
  dataPoint: DataPoint,
  history: HistoryWindow,
): EvaluatorResult;
```

`evaluateRule` is **pure**. No `Date.now()`; the caller passes `now` via `dataPoint` or via `history` for the inactivity tick. No DB calls. No realtime side-effects. This is what makes the preview-vs-live parity test possible.

`cooldown.ts` exports:

```ts
export function cooldownSeconds(rule: AlertRule): number; // resolves override or default per CROSS_CUTTING §3
export function withinCooldown(rule: AlertRule, lastFiredAt: string | null, now: string): boolean;
```

## Migrations

Filenames follow `YYYYMMDDHHMMSS_<slug>.sql` per [CROSS_CUTTING.md §12](../CROSS_CUTTING.md#12-migrations-discipline). Timestamps are sequential.

### `<ts>_events_table.sql`

Adds the Phase 4 `events` table that PHASES.md flags as new in this phase (it was not in the foundational 12).

```sql
create table public.events (
  id uuid primary key default gen_random_uuid(),
  patient_id uuid not null references public.patients(id) on delete cascade,
  device_id uuid references public.devices(id) on delete set null,
  occurred_at timestamptz not null,
  type text not null,
  payload jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now()
);
create index events_patient_occurred_idx on public.events(patient_id, occurred_at desc);

alter table public.events enable row level security;
create policy events_allocated_read on public.events
  for select using (public.is_caregiver_for(patient_id));
-- inserts service-role only; no insert policy → RLS denies authenticated callers.

alter publication supabase_realtime add table public.events;
```

Description: introduces the events ingestion target so the bridge has a place to persist `EventMessage` rows; rules engine reads from this table for the fall rule type. **Rollback**: `drop table public.events cascade;` (deletes any persisted events).

### `<ts>_alert_rules_writes.sql`

Adds RLS write policies for `alert_rules`. Read policy already exists from foundation.

```sql
create policy alert_rules_allocated_insert on public.alert_rules
  for insert with check (public.is_caregiver_for(patient_id));

create policy alert_rules_allocated_update on public.alert_rules
  for update using (public.is_caregiver_for(patient_id))
  with check (public.is_caregiver_for(patient_id));

create policy alert_rules_allocated_delete on public.alert_rules
  for delete using (public.is_caregiver_for(patient_id));
```

Description: lets allocated caregivers manage rules for their patient via the JS client; service role bypasses RLS for engine reads. **Rollback**: drop the three policies.

### `<ts>_acknowledge_alert_rpc.sql`

`SECURITY DEFINER` RPC the F12 ack button calls. Defined here because F11 owns the rules + alerts surface as a whole.

```sql
create or replace function public.acknowledge_alert(p_alert_id uuid)
returns public.alerts
language plpgsql
security definer
set search_path = public
as $$
declare
  v_row public.alerts;
begin
  select * into v_row from public.alerts where id = p_alert_id;
  if not found then
    raise exception 'alert not found' using errcode = 'P0002';
  end if;
  if not public.is_caregiver_for(v_row.patient_id) then
    raise exception 'not allocated to patient' using errcode = '42501';
  end if;
  if v_row.acknowledged_at is not null then
    return v_row; -- idempotent
  end if;
  update public.alerts
    set acknowledged_at = now(),
        ack_by_caregiver_id = auth.uid()
   where id = p_alert_id
   returning * into v_row;
  return v_row;
end;
$$;

revoke all on function public.acknowledge_alert(uuid) from public;
grant execute on function public.acknowledge_alert(uuid) to authenticated;
```

Description: caregivers ack via RPC rather than a column-level RLS policy on `alerts` (the column-level approach is brittle — see [CROSS_CUTTING.md §1](../CROSS_CUTTING.md#1-rls-write-policies)). Idempotent. **Rollback**: `drop function public.acknowledge_alert(uuid);`.

### `<ts>_pg_cron_inactivity.sql`

Enables the `pg_cron` extension and schedules `inactivity_scan` every 60 s.

```sql
create extension if not exists pg_cron;

select cron.schedule(
  'inactivity_scan_every_minute',
  '* * * * *',
  $$select net.http_post(
      url := current_setting('app.inactivity_scan_url'),
      headers := jsonb_build_object('content-type', 'application/json',
                                    'authorization', 'Bearer ' || current_setting('app.service_role_key'))
   );$$
);
```

Description: schedules a per-minute POST to the `inactivity_scan` Edge Function URL. The URL and the service-role key are set via `alter database ... set app.<key> = ...` in deployment, not committed. **Rollback**: `select cron.unschedule('inactivity_scan_every_minute'); drop extension pg_cron;`. **If pg_cron is unavailable on the chosen Supabase plan**, omit this migration and document the external scheduler config in the PR description plus `.env.example`.

## RPCs / edge function changes

- **`acknowledge_alert(alert_id uuid)`** — new RPC (above). F12's `AckButton` calls this.
- **`rules_engine`** — replace stub. Triggered by DB webhook on INSERT into `sensor_readings`, `position_estimates`, and `events`. For each enabled rule on the patient whose type matches the incoming source, build a `DataPoint`, load the smallest sufficient `HistoryWindow` (only what the evaluator needs — e.g. last `dwell_seconds` of position estimates for zone), call `evaluateRule`, check cooldown via `withinCooldown` against the last unacked alert for the (patient, rule), insert into `alerts` if firing.
- **`inactivity_scan`** — new HTTP-triggered Edge Function. Invoked by `pg_cron` (or external scheduler). Iterates enabled inactivity rules; for each, builds a `tick` `DataPoint` with `at = now()`, loads the last position estimate, calls `evaluateRule`, applies cooldown, inserts. Uses the service-role client.

Both functions log structured JSON per [CROSS_CUTTING.md §9](../CROSS_CUTTING.md#9-logging--observability) — `level`, `msg`, `patient_id`, `rule_id`, `decision` (`fire` / `cooldown_suppressed` / `no_match`).

## Tests

- **Evaluator parity test** (`packages/shared/src/rules/__tests__/parity.test.ts`) — the canary from [CROSS_CUTTING.md §10](../CROSS_CUTTING.md#10-rule-evaluator-location). Seeds 24 h of synthetic data; runs the live engine path (via the HTTP entry of `rules_engine` against a seeded local Supabase) and runs the preview path (direct `evaluateRule` calls in-process); asserts the resulting alert sets are array-equal on `(rule_id, fired_at, severity, context)`. **Required-or-CI-fails.**
- **Per rule type** (`evaluate.test.ts`) — for each of the four V1 types: positive case (matching data → `fire: true` with the expected severity and context), negative case (boundary data → `fire: false`), cooldown case (two matching data points within the cooldown window — assert the second is suppressed by `withinCooldown`).
- **Dispatch test** (`rules_engine/__tests__/dispatch.test.ts`) — drive the function's HTTP handler with a webhook payload for each source table; assert the right rule type evaluator is invoked and an alert row appears.
- **Scan test** (`inactivity_scan/__tests__/scan.test.ts`) — seed a patient whose last `position_estimates` row is older than the rule threshold; invoke the function; assert one alert row.
- **TST-09 alert delivery reliability** — for each of the four rule types, drive 10 trigger samples through the full chain; assert all 10 surface in `alerts` and the wall-clock latency from trigger to row insert is within budget (zone/vitals/fall: ≤ 2 s; inactivity: ≤ 65 s due to the cron tick).
- **TST-08 geofence breach** — synthetic position replay walks across the polygon edge; assert the zone alert fires within 5 s.
- **Cooldown integration test** — fire 5 vitals triggers within 30 s; assert exactly one alert row.
- **Rule update propagation** — update a rule's threshold; within 30 s, the new threshold is in effect (drive a triggering data point that was below the old threshold and would not have fired, but is above the new one).
- **RLS denial** (`apps/web/src/test/rules-rls.test.ts`) — caregiver A's JS client cannot insert `alert_rules` rows naming caregiver B's patient.

## Risks

- **`pg_cron` may not be on the project's Supabase plan.** Confirm before locking the inactivity migration. Fallback path: an external scheduler (Vercel cron, GitHub Actions on `* * * * *`) hits the `inactivity_scan` Edge Function URL with a bearer token. Document the chosen path in the PR description and `.env.example`.
- **False positive rate** is graded under EV-05. Defaults will be noisy on real data. Mitigation: ship reasonable defaults (vitals ranges from common clinical guidance; inactivity 60 min daytime / disabled overnight); the live preview lets the caregiver tune before saving. Honest reporting in the evaluation report is the contract.
- **Cooldown edge case — rule re-enabled after disable.** A rule that was disabled for several hours then re-enabled should not be silenced because a stale alert sits within the cooldown window. **Recommendation**: disabling a rule resets its cooldown state — treat the engine's cooldown query as `select max(fired_at) from alerts where patient_id = $1 and rule_id = $2 and acknowledged_at is null and fired_at >= alert_rules.updated_at`. The `updated_at` bump on enable makes the cooldown ignore older firings. Document this once in the function's source comment and once here.
- **Preview-versus-live drift.** If `evaluate.ts` ever stops being the only path, caregivers can't trust the preview. The parity test guards against this; if it goes flaky, fix the divergence rather than skipping the test.
- **Polygon authoring is shared with [F9](./F9.md).** F9 owns the polygon-draw UI; F11's `ZoneRuleCard` consumes it. If F9 ships after F11 starts, stub the polygon picker with a textarea taking JSON until the picker lands.
- **Preview cost on large history.** 24 h of `sensor_readings` at 1 Hz is 86,400 rows per patient. The preview runs the evaluator in the browser. Mitigation: server-side pagination (load in 1 h chunks) and decimate if rendering becomes a bottleneck. Acceptable for prototype scope.
- **Cron clock skew.** `pg_cron` ticks every minute on the database clock. Inactivity rules that are tuned to the minute may fire one tick later than expected. Acceptable; the inactivity rule type is not minute-precision-critical.
