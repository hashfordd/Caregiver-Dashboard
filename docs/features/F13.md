# F13 — Activity History & Logs

## Goal

A caregiver opens a patient's dashboard, picks "History", and replays the last
N hours of indoor movement on the same Fabric.js canvas the live marker uses;
scrubs vitals over a selectable date range on a Recharts line chart; filters
the alert history by date / severity / type; and exports vitals or positions
to CSV for handover or clinical review. F13 is the only V1 feature in
[Phase 5](../PHASES.md#phase-5--polish); the rest of Phase 5 is accessibility
(UI-28), mobile responsiveness (UI-29), and demo prep (DOC-10..12), tracked at
the workstream level rather than as feature docs.

## Acceptance criteria

Restated from the build spec for F13 and the Phase 5 exit criteria in
[../PHASES.md](../PHASES.md#phase-5--polish):

- The caregiver can replay 1 hour of indoor movement at 10× speed against the
  saved floor plan; the playback completes in 6 minutes wall-clock without
  dropped frames.
- The caregiver can scrub a vitals chart across a selectable date range
  (presets: last 1 h / 6 h / 24 h / 7 d / custom). Lines render for HR, SpO2,
  and temperature using the caregiver's `°C` / `°F` preference from F1.
- The caregiver can filter the alert history by date range, severity
  (`info` / `warn` / `critical`), and rule type (`vitals` / `zone` / `fall` /
  `inactivity`).
- The caregiver can export the last 24 h of vitals to CSV; the file downloads
  cleanly and opens in a spreadsheet without quoting / encoding errors.
- All four states (loading / empty / error / stale) are presented per
  [../CROSS_CUTTING.md §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states).

The Phase 5 acceptance text in [../PHASES.md](../PHASES.md#phase-5--polish)
captures the load-bearing pair: "replay 1 h at 10× speed; CSV export of last
24 h vitals downloads cleanly." All other criteria above are derived from the
F13 spec and are restated here so an implementer does not need to cross
multiple docs to know when the feature is done.

## Dependencies

**Hard prerequisites**

- Phases 1–4 closed. F13 reads from `sensor_readings`, `position_estimates`,
  `alerts`, and `alert_rules` — every row F13 surfaces was written by
  earlier phases. F13 is read-only against these tables.
- F5 (floor plan) — replay renders on the saved floor plan canvas; without a
  floor plan there is nothing to replay against. The history tab surfaces the
  empty state from [../CROSS_CUTTING.md §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states)
  with a "Set up a floor plan first →" CTA when no `floor_plans` row exists.
- F4 (sensor cards) — the temperature unit preference (`°C` / `°F`) is the
  one F4 honours; F13's vitals chart imports the same formatter.
- F12 (alert feed) — severity colour map is fixed there; the alert history
  filter chips reuse the same tokens per
  [../CROSS_CUTTING.md §15](../CROSS_CUTTING.md#15-brand-consistency).

**Cross-cutting**

- [../CROSS_CUTTING.md §8 — Telemetry retention](../CROSS_CUTTING.md#8-telemetry-retention)
  defines the long-term aggregate shape (`sensor_readings_1m`,
  `position_estimates_1m`). F13's UI works on raw rows for prototype windows;
  see [Migrations](#migrations) for the conditional retention compactor that
  is only shipped if TST-10 surfaces volume issues.
- [../CROSS_CUTTING.md §16 — Performance budgets](../CROSS_CUTTING.md#16-performance-budgets)
  caps the 24 h history page at 2 s initial paint. The Recharts bundle
  (~80 KB gzipped) and Fabric.js (already installed by F5) push the budget;
  F13 lazy-loads the History tab so the bundle is not requested until the
  caregiver opens the tab.
- [../CROSS_CUTTING.md §7 — Realtime patterns](../CROSS_CUTTING.md#7-realtime-patterns)
  separates live data from server state. F13 is **server state only** — no
  realtime subscriptions. React Query owns every fetch.
- [../CROSS_CUTTING.md §14 — Accessibility floor](../CROSS_CUTTING.md#14-accessibility-floor)
  applies; canvas accessibility for replay is V2 — F13 ships keyboard
  shortcuts on the scrubber (space = play/pause, ← / → = step, shift + ← / → =
  jump 10 s).

**Soft prerequisites**

- TST-10 (60-minute continuous run) is the Phase 5 entry condition. F13's
  worst-case window depends on what TST-10 surfaces; if memory growth or
  query latency are red, the retention compactor migration ships alongside
  F13 rather than after.

**What this feature unblocks**

- Nothing further in V1. F13 closes the V1 feature surface. The CSV export
  shape is the seam V2 ML training-data extraction will lean on; documenting
  a stable column order is part of this feature's scope (see
  [Contracts in `packages/shared`](#contracts-in-packagesshared)).

## Contracts in `packages/shared`

F13 is a consumer of existing types — `sensor_readings` rows,
`position_estimates` rows, `alerts` rows. No new schemas. The one stable
contract F13 introduces is the **CSV column order**, which V2 ML training
data extraction will read from. Document it once and treat it as a wire
format:

- Vitals CSV header (in this order):
  `recorded_at,hr_bpm,spo2_pct,temp_c`
  Notes: `recorded_at` is the ISO 8601 UTC timestamp from the row; numeric
  cells are unquoted; null cells are an empty field (two adjacent commas);
  `temp_c` always exports in Celsius regardless of the caregiver's display
  preference, so the file is locale-independent.
- Positions CSV header (in this order):
  `recorded_at,mode,x_canvas,y_canvas,lat,lng,confidence`
  Notes: `mode` is the `position_mode` enum value as a string;
  `(x_canvas, y_canvas)` are populated for indoor rows, `(lat, lng)` for
  outdoor rows; the unused pair is empty cells.
- Alerts CSV header (in this order):
  `fired_at,severity,rule_type,acknowledged_at,ack_by_caregiver_id`
  Notes: `acknowledged_at` and `ack_by_caregiver_id` are empty for unacked
  rows. (The alert export is not in the Phase 5 acceptance criteria but
  ships behind the same export sub-tab — costs nothing to include.)

The serialiser quotes any cell containing a comma, a double quote, a newline,
or a leading / trailing space, per RFC 4180. Embedded double quotes are
doubled (`"foo""bar"`). The unit test in `csv.test.ts` covers each of these.

No `packages/shared/src/csv/` module is introduced; the serialiser is
local to `apps/web` because it is consumed only by the dashboard. If V2
needs the same shape on a server-side export path, lift the file to
`packages/shared` then.

## Migrations

**None required for V1.** F13's UI works on raw rows from
`sensor_readings`, `position_estimates`, `alerts` per
[../CROSS_CUTTING.md §8](../CROSS_CUTTING.md#8-telemetry-retention).

**Conditional addition flagged for Phase 5**: if TST-10's 60-minute
continuous run reveals query latency > 2 s for the 24 h history page or
memory growth on the dashboard tab, ship the retention compactor as part of
F13. The migration shape is additive and does not require backfill of
existing rows:

- `<ts>_sensor_readings_1m_aggregate.sql` — new (conditional). Creates
  `public.sensor_readings_1m` with columns
  `(patient_id uuid, bucket_start timestamptz, hr_avg numeric, hr_min numeric,
hr_max numeric, spo2_avg numeric, spo2_min numeric, spo2_max numeric,
temp_c_avg numeric, sample_count integer)`, primary key
  `(patient_id, bucket_start)`, RLS read-scoped via `is_caregiver_for(patient_id)`.
- `<ts>_position_estimates_1m_aggregate.sql` — new (conditional). Creates
  `public.position_estimates_1m` with columns
  `(patient_id uuid, bucket_start timestamptz, x_canvas_avg numeric,
y_canvas_avg numeric, lat_avg numeric, lng_avg numeric,
confidence_avg numeric, mode public.position_mode, sample_count integer)`,
  primary key `(patient_id, bucket_start)`, RLS read-scoped.
- `apps/edge/functions/retention_compactor/index.ts` — new (conditional).
  Scheduled Edge Function (pg_cron, nightly) that runs the SQL job
  documented in [../CROSS_CUTTING.md §8](../CROSS_CUTTING.md#8-telemetry-retention)
  for the previous day's rows. Service-role write per
  [../CROSS_CUTTING.md §1](../CROSS_CUTTING.md#1-rls-write-policies).

Rollback note for both aggregate tables: drop table — additive, no source
data is mutated by the compactor (the original rows in `sensor_readings` /
`position_estimates` are read-only inputs). The compactor itself is
idempotent on `(patient_id, bucket_start)` via `on conflict do update`.

**Decision rule**: do not pre-emptively ship the compactor. The acceptance
in [../CROSS_CUTTING.md §8](../CROSS_CUTTING.md#8-telemetry-retention) is
"the migration leaves room for retention without rework," which the
foundation already satisfies. Only land the compactor if measured volume in
TST-10 demands it.

## RPCs / edge function changes

**None required for V1.** F13 reads the raw tables via the standard
Supabase client; the existing RLS read policies on `sensor_readings`,
`position_estimates`, and `alerts` (foundation migration lines 251–261)
gate access correctly. Per-row latency is dominated by the network round
trip, not RLS evaluation.

**Considered and deferred**: a server-side decimating RPC
`select_history(patient_id, from, to, max_points)` that returns at most
`max_points` rows by sampling every Nth via `row_number() % stride = 0`.
The trade-off:

- Pro: reduces 24 h × 1 Hz = 86,400 rows (~6–8 MB JSON over the wire) to a
  configurable target like 2,000 rows (~200 KB). Closes the 2 s budget for
  the 24 h vitals chart.
- Con: chart fidelity drops; replay correctness is harmed (the replay
  contract is "dots in time order"; skipping rows is a UX change). Adds an
  RPC + a migration that may not be needed.

**Recommendation**: ship simple paginated fetches first, with the default
replay window pinned to 1 h (3,600 rows) and the default vitals chart
window pinned to 6 h (21,600 rows). Both fit comfortably under the budget.
Only add the RPC if the 24 h export view is sluggish in TST-10. Do not
pre-optimise.

## Tests

Vitest, run via `npm run test --workspace apps/web`. Per
[../CROSS_CUTTING.md §13](../CROSS_CUTTING.md#13-testing-strategy).

- **CSV serialisation** (`csv.test.ts`). Pure-function unit tests on
  `vitalsRowsToCsv` and `positionRowsToCsv`:

  1. Header row matches the documented order.
  2. Numeric cells unquoted; null cells are empty fields.
  3. String cells containing a comma are quoted: `"a,b"`.
  4. String cells containing a double quote double-up the quote: `"a""b"`.
  5. String cells containing a newline are quoted and the newline is
     preserved.
  6. Empty input produces a header-only file (one line, trailing CRLF).
  7. CRLF line terminator per RFC 4180.

- **Replay correctness** (`ReplayScrubber.test.tsx`). Given a fixture of 60
  `position_estimates` rows at 1 Hz over 60 seconds:

  1. Scrubber renders the dots in time order at 10× speed.
  2. At t = 30 s of playback wall-clock, the scrubber index corresponds to
     row 30 within tolerance.
  3. After the scrubber crosses a row, that row's dot is removed from the
     canvas (assert via `canvas.getObjects().length` or via an explicit
     mock on `canvas.remove`). This is the memory-leak guard from
     [Risks](#risks).

- **Alert history filter** (`AlertHistoryFilter.test.tsx`). Given a fixture
  of 30 alerts spanning all three severities and four rule types:

  1. No filters → all 30 rows present.
  2. Severity = `critical` only → only the critical-severity rows.
  3. Rule type = `fall` only → only fall-rule rows.
  4. Severity = `warn` and rule type = `vitals` → intersection.
  5. Date range narrowed to half the corpus → expected subset.

- **Manual smoke (the F13 acceptance gates)**:
  1. Replay 1 h at 10× speed runs in 6 minutes wall-clock without UI lag
     (frame budget < 16 ms per tick on the demo laptop).
  2. CSV export of last 24 h vitals downloads in under 5 s and opens
     cleanly in a spreadsheet.
  3. Vitals chart selectable date range works across all four presets and
     the custom range.
  4. Alert history filter combines severity / rule type / date predictably.

The replay correctness test is the Phase 5 verification gate item from
[../PHASES.md](../PHASES.md#phase-5--polish) ("Replay 1 h at 10×").

## Risks

- **24 h export volume**. 24 h × 1 Hz = 86,400 rows ≈ 6–8 MB of JSON over
  the wire for vitals; same for positions. Mitigation: window-based
  fetches; default vitals chart window pinned to 6 h; 24 h only on
  explicit user request via the export sub-tab. Server-side decimation via
  the `select_history` RPC (see [RPCs / edge function changes](#rpcs--edge-function-changes))
  is the next lever and is deferred until measurement justifies it.
- **Recharts bundle weight**. Roughly 80 KB gzipped. Per
  [../CROSS_CUTTING.md §16](../CROSS_CUTTING.md#16-performance-budgets) the
  initial JS bundle budget is < 200 KB; Recharts pushes it. Mitigation:
  the History tab is lazy-loaded via
  `const HistoryTab = lazy(() => import('./features/history/HistoryTab'))`
  so the chunk is requested only when the caregiver opens the tab. Verify
  in the network tab during the manual smoke.
- **Memory leaks during long replay**. Fabric.js objects accumulate if the
  replay renders a dot per frame and never removes them. For a 1 h replay
  at 1 Hz that is 3,600 dots, which is fine; for a 24 h replay at 1 Hz
  that is 86,400 dots, which is not. Mitigation: the scrubber renders only
  the dot at the current playback position plus a configurable trail
  (default 60 s = last 60 dots); dots whose timestamp precedes the trail
  window are explicitly removed via `canvas.remove(dot)`. The unit test in
  `ReplayScrubber.test.tsx` asserts the removal. Do not rely on garbage
  collection — Fabric holds references in its object array.
- **Canvas accessibility for replay**. Fabric canvases are not natively
  keyboard-navigable. Per
  [../CROSS_CUTTING.md §14](../CROSS_CUTTING.md#14-accessibility-floor)
  full canvas a11y is V2. F13 minimum: scrubber keyboard shortcuts
  (space = play/pause, ← / → = step one row, shift + ← / → = jump 10 rows);
  scrubber rendered as a labelled `<input type="range">` so screen readers
  announce position; canvas content read aloud is V2.
- **Timezone surprises in CSV**. `recorded_at` is exported as ISO 8601 UTC
  with `Z` suffix per
  [../CROSS_CUTTING.md §5](../CROSS_CUTTING.md#5-time-handling). Spreadsheet
  software may auto-convert to local time on import; this is acceptable
  for V1 and is documented in the export sub-tab help text ("timestamps
  are UTC; your spreadsheet may convert to local time on open").
- **Alert filter combinatorics**. Three filter dimensions × N values each
  is small enough to evaluate client-side. Mitigation: filter the React
  Query result client-side rather than re-fetching per filter combination;
  the date range is the only predicate sent to the server.
- **Stale data during a long-open History tab**. F13 is server state, not
  realtime. If the caregiver leaves the tab open for hours, new alerts /
  vitals will not appear without a refetch. Mitigation: React Query's
  `staleTime: 30_000` plus a manual "Refresh" button on the tab header.
  Realtime subscription is deliberately not used here per
  [../CROSS_CUTTING.md §7](../CROSS_CUTTING.md#7-realtime-patterns).
