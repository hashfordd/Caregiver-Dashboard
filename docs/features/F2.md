# F2 — Patient Roster & Allocation

## Goal

A signed-in caregiver lands on a roster that lists every patient they are
allocated to (via `caregiver_patient`), can create a new patient and be
auto-allocated to it in a single transaction, and can click through to the
patient detail dashboard. Empty roster surfaces a clear "create your first
patient" state. Closing F2 means the dashboard has its primary navigation
spine: every downstream feature (F3, F4, F10, and Phase 2+ work) is reached
by clicking a row on this list.

## Acceptance criteria

Restated from the spec and Phase 1 critical path:

- A `/patients` route renders all patients allocated to the signed-in
  caregiver, ordered by `full_name` ascending. Empty result renders a
  friendly empty state with a "Create patient" CTA per
  [CROSS_CUTTING §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states).
- A "Create patient" dialog or sub-route captures `full_name` (required),
  `dob` (optional date), `notes` (optional text). On submit, the
  `create_patient_with_allocation` RPC (shipped by [F1.md](./F1.md))
  inserts the patient and the `caregiver_patient` allocation atomically.
  After success, the dialog closes and the new row appears at the top of
  the list (or in alphabetical order on next refresh).
- Clicking a row navigates to `/patients/:id` (the F3 shell). If F3 has not
  yet shipped, the route renders a placeholder Card — F2 ships the
  navigation, F3 ships the body.
- An RLS denial test (extended from F1's file) confirms that caregiver A
  cannot read patient B from the roster query.
- The post-login landing page at `/` redirects to `/patients` (replaces
  `LandingPage` as the post-login default).

No spec corrections found for F2. The Phase 1 risk note about "family
caregivers without create permission" is **out of scope** for F2 — every
caregiver in V1 may create patients; an invite/grant path is deferred.
Document in BACKLOG.

## Dependencies

**Hard prerequisites**

- [F1.md](./F1.md) — auth boundary, write policies on `patients` and
  `caregiver_patient`, and the `create_patient_with_allocation` RPC must
  ship before F2 can mutate. (F1 ships the RPC because F1 owns the policy
  surface; F2 consumes it.)
- [CROSS_CUTTING §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states)
  — loading skeleton + empty state + error retry are required on the
  roster list.
- [CROSS_CUTTING §7 — Realtime patterns](../CROSS_CUTTING.md#7-realtime-patterns)
  — roster is **server state**, not live data; React Query manages the
  cache.

**Soft prerequisites**

- None at the level of F2's own work; the roster does not need realtime in
  V1. (Adding realtime invalidation when a peer caregiver allocates a new
  patient is V2.)

**What this feature unblocks**

- [F3.md](./F3.md) — the dashboard shell mounts at `/patients/:id` and is
  reached from the roster row click.
- [F10.md](./F10.md) — device pairing happens inside the patient detail
  view, so reaching the patient is required.

## Contracts in `packages/shared`

- `packages/shared/src/db/patients.ts` — new; exports the read-shape and
  the create-shape for patients:

  ```ts
  import { z } from 'zod';
  export const Patient = z.object({
    id: z.string().uuid(),
    full_name: z.string().min(1),
    dob: z.string().date().nullable(),
    notes: z.string().nullable(),
    primary_caregiver_id: z.string().uuid().nullable(),
    created_at: z.string().datetime(),
  });
  export type Patient = z.infer<typeof Patient>;

  export const CreatePatientInput = z.object({
    full_name: z.string().min(1).max(120),
    dob: z.string().date().nullable().optional(),
    notes: z.string().max(2000).nullable().optional(),
  });
  export type CreatePatientInput = z.infer<typeof CreatePatientInput>;
  ```

## Risks

- **Two-write race during creation**. The RPC is atomic, so this is
  closed at the DB layer. Confirm in the RLS test that the
  `caregiver_patient` row is committed before the function returns.
- **Stale roster on multi-tab usage**. If a caregiver creates a
  patient in tab A, tab B's roster doesn't auto-refresh. Acceptable
  for V1; document. If escalation, subscribe `usePatientStream` is
  the wrong tool (it's per-patient); a peer-level subscription on
  `caregiver_patient` is V2 work.
- **Patient delete UX**. Out of scope for F2; the F1 policy permits
  it but no UI ships. If a caregiver inserts a typo'd patient, they
  can either re-create or wait for V2's edit/delete UI. Document in
  BACKLOG.
- **Avatar / photo upload**. Out of scope; storage buckets aren't
  provisioned until F5.
- **Roster pagination**. V1 caregivers will have ≤10 patients;
  unbounded for V1. If a caregiver hits 100+, the client query needs
  paging. V2.
