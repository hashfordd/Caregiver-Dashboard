# F1 — Authentication & Caregiver Account

## Goal

A caregiver can create their own account from the public site, choose whether
they are a `professional` or `family` caregiver during signup, sign in
afterwards via password or magic link, and view and edit their own profile
(name, email, role, temperature unit preference). Closing F1 means the
authentication boundary of the dashboard is fully wired: the foundation trigger
that materialises a `caregivers` row from the `auth.users` insert is exercised
end-to-end by a real signup form, the read-scoping RLS that already shipped is
joined by the missing **write** policies, and a CI grep rule prevents the
service-role key from ever leaking into the browser bundle. After F1 lands,
every other Phase 1 feature has a real signed-in subject to reason about.

## Acceptance criteria

Restated from the build spec and Phase 1 entry/exit:

- A new caregiver can sign up at `/signup` with email, password, full name, and
  role (`professional` | `family`). Submitting the form results in a row in
  `public.caregivers` (via `handle_new_user`) with `role` and `full_name` taken
  from `raw_user_meta_data` — both already wired in the foundation migration.
- Sign-in at `/login` works for password and magic-link modes (already shipped).
- Sign-out clears the session and returns the caregiver to `/login`.
- A `/profile` route reads the caregiver's row from `public.caregivers` and
  permits self-update of `full_name` and `role`. Email is read-only (it lives
  on `auth.users` and updating it is out of scope for V1 — defer to BACKLOG).
- A self-update from the profile page is denied for any other caregiver's row
  (RLS test, see [Tests](#tests)).
- Creating a patient (F2) inserts both the `patients` row and the corresponding
  `caregiver_patient` row in one transaction; the policies enabling that are
  this feature's deliverable, even though the UI lives in F2.
- The CI pipeline fails if `SUPABASE_SERVICE_ROLE_KEY` (or any string matching
  `SERVICE_ROLE`) appears anywhere under `apps/web/`.

No spec corrections found for F1. The Phase 1 entry note about the
`admin@bizzieapp.com` test user is a smoke fixture, not an acceptance
criterion — the feature passes when the **signup form** can produce a working
caregiver, regardless of any seeded user.

## Migrations

- `supabase/migrations/<ts>_f1_caregiver_write_surface.sql` — new. Creates:

  - `caregivers_self_update` (insert is trigger-only; update is already
    self-scoped in the foundation migration — verify the policy name matches
    and only **add** an explicit `caregivers_no_insert` deny-by-default if
    necessary; the foundation migration's existing `caregivers_self_update`
    suffices, so this migration **does not** redeclare it). Confirm in review.
  - `patients_self_insert` — `with check (primary_caregiver_id = auth.uid())`.
    The creator must set themselves as primary caregiver on insert.
  - `patients_allocated_update` — `using/with check (is_caregiver_for(id))`.
  - `patients_allocated_delete` — `using (is_caregiver_for(id))`.
  - `caregiver_patient_self_insert` — `with check (caregiver_id = auth.uid()
or is_caregiver_for(patient_id))`. The first clause covers
    creator-auto-allocate; the second covers an already-allocated caregiver
    inviting a peer. The RPC bypass discussed below avoids the chicken-and-egg
    problem on a brand-new patient where neither side of the OR is yet true
    via the user's session, by running with `security definer`.
  - `caregiver_patient_self_delete` — `using (caregiver_id = auth.uid())`.
    A caregiver can remove their own allocation; removing someone else's is
    out of scope.

  Rollback note: drop policies (additive-no-rollback for table data; safe to
  `drop policy ... on public.<table>` per policy).

- `supabase/migrations/<ts>_f1_create_patient_with_allocation.sql` — new.
  Defines:
  ```sql
  create or replace function public.create_patient_with_allocation(
    p_full_name text,
    p_dob date default null,
    p_notes text default null
  ) returns public.patients
  language plpgsql security definer set search_path = public as $$
  declare
    new_patient public.patients;
    caller uuid := auth.uid();
  begin
    if caller is null then
      raise exception 'auth required';
    end if;
    insert into public.patients (full_name, dob, notes, primary_caregiver_id)
    values (p_full_name, p_dob, p_notes, caller)
    returning * into new_patient;
    insert into public.caregiver_patient (caregiver_id, patient_id, role)
    values (caller, new_patient.id, 'creator');
    return new_patient;
  end$$;
  grant execute on function public.create_patient_with_allocation(text, date, text)
    to authenticated;
  ```
  Rollback note: drop function.

## RPCs / edge function changes

- `create_patient_with_allocation(full_name, dob, notes) returns patients` —
  `SECURITY DEFINER`. Inserts the patient with `primary_caregiver_id = auth.uid()`
  and a paired `caregiver_patient` row in the same transaction. **Why an RPC
  rather than client-side two-step**: a client-side two-step (insert into
  `patients` first, then `caregiver_patient`) is exposed to a half-allocated
  state if the second insert fails (network drop, RLS surprise, race).
  Wrapping both in a single `SECURITY DEFINER` function gives transactional
  atomicity, hides the policy choreography from the client, and centralises
  the "creator gets allocated" business rule in one place. The trade-off is
  a small expansion of the privileged surface — mitigated by the explicit
  `auth.uid()` null check, the implicit search-path lock, and the narrow
  parameter list.
- No edge function changes for F1. The bridge is unaffected.

## Risks

- **Signup verification disabled in dev, enabled in prod**. Supabase
  defaults differ. The form must handle the case where `signUp` returns a
  user without a session (verification email pending). Mitigation: check
  `data.session` in the response; if null, route to a "check your inbox"
  state instead of `/`.
- **`role` enum drift**. If a future migration adds a third role
  (e.g. `admin`), the signup select goes stale. Mitigation: import
  `CaregiverRole` from `packages/shared/src/db/caregivers.ts` in the select
  options so the enum is single-sourced. A failing typecheck on schema
  change is the desired outcome.
- **Service-role key leakage**. Mitigated by the CI grep step. Edge case:
  the rule must not match against `apps/web/.env.example` or fixture files
  that legitimately mention the variable name in commentary. If false
  positives appear, narrow the grep to `*.ts`, `*.tsx`, `*.js`, `*.json`,
  `*.html`. Acceptable for V1.
- **Policy churn during F2 development**. F2 will exercise
  `create_patient_with_allocation` heavily. Lock the RPC signature in F1
  and freeze it through Phase 1; if F2 needs additional fields (e.g.
  `dob`), they're already in the parameter list as nullable.
- **Email change deferred**. Caregivers who want to change their email need
  to re-create the account in V1. Add to BACKLOG.
