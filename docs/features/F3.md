# F3 — Patient Detail Dashboard (shell)

## Goal

A caregiver who clicks a roster row arrives at a patient detail page that
identifies the patient (header with name, age, allocated caregiver list,
realtime status pill) and exposes a tabbed body where each tab is a feature
surface. In Phase 1 the only populated tab is **Live**, which hosts the F4
sensor cards and the F10 device pairing CTA. Other tabs (Place, History,
Alerts, Settings) render a placeholder until their owning features land.

The shell's load-bearing job is to **mount the realtime stream once at this
level via a context** so F4 and every later tab consumes the same
subscription. If F4 mounted `usePatientStream` directly inside its component,
switching tabs would tear the channel down and re-establish it; sparkline
state would be discarded. F3's context is the seam that prevents that.

## Acceptance criteria

Restated from spec and Phase 1 critical path:

- A `/patients/:id` route renders the patient header (name, age computed
  from `dob`, `primary_caregiver_id` resolved to a caregiver name if
  available) and a tab strip below it.
- The tab strip exposes five tabs: **Live**, **Place**, **History**,
  **Alerts**, **Settings**. Only **Live** has content in Phase 1; the
  others render a "Coming in Phase N" placeholder Card.
- The shell mounts `usePatientStream(id, callbacks)` exactly once via a
  React Context (`PatientStreamContext`). Switching tabs does not
  re-subscribe (verifiable by a render-count assertion in tests; or by
  observing in dev tools that the channel ID is stable across tab
  switches).
- A connection-status pill in the header displays the realtime channel
  state: `subscribed`, `disconnected`, `error`. Source of truth is the
  hook's `status` (see [Risks](#risks) — the foundation hook does not
  expose `status` yet; F3 widens the hook signature).
- Resolution of an unknown `:id` (no allocated patient with that ID) renders
  a "Patient not found" empty state and a back-link to `/patients`. RLS
  ensures the query returns empty rather than denying outright.

No spec corrections found for F3. The dashboard tab list ("Live, Place,
History, Alerts, Settings") is not literally enumerated in PHASES.md but
follows from the Phase 2/3/4/5 feature mapping; if a future feature needs a
sixth tab, add it then.

## Files to create or modify

### Frontend

- `apps/web/src/features/patients/PatientDetailPage.tsx` — new; the
  `/patients/:id` route component. Reads `id` from `useParams`, fetches
  the patient row + the patient's allocated caregivers via React Query.
  Wraps its body in `<PatientStreamProvider patientId={id}>`. Renders
  `<PatientHeader>` and `<PatientTabs>`.
- `apps/web/src/features/patients/PatientStreamContext.tsx` — new; a
  React context exposing the running stream state plus `lastSeen` per
  channel kind. Internally calls `usePatientStream` once. Shape:
  ```ts
  interface PatientStreamContextValue {
    patientId: string;
    status: 'idle' | 'subscribed' | 'disconnected' | 'error';
    lastSeen: { sensor: number | null; position: number | null; alert: number | null };
    onSensorReading: (cb: (row: SensorReadingRow) => void) => () => void;
    onPositionEstimate: (cb: (row: PositionEstimateRow) => void) => () => void;
    onAlert: (cb: (row: AlertRow) => void) => () => void;
  }
  ```
  The context implements a pub/sub fanout — `usePatientStream` writes
  rows to internal subscriber lists, components register callbacks and
  receive a teardown closure. This pattern lets multiple components on
  the same tab consume the same stream without each opening a channel.
- `apps/web/src/features/patients/PatientHeader.tsx` — new; renders
  patient name, computed age, allocated caregiver chips, and the
  `<ConnectionStatusPill>` driven by the context's `status`.
- `apps/web/src/features/patients/PatientTabs.tsx` — new; a Radix Tabs
  primitive. Tabs render lazy children — only the active tab mounts.
  Tab keys: `live`, `place`, `history`, `alerts`, `settings`. URL state:
  the tab key is reflected as `?tab=<key>` in the search params so deep
  links work. Default `live` if missing.
- `apps/web/src/features/patients/tabs/LiveTab.tsx` — new; ships as a
  composition slot in F3. F3 lands it as a placeholder (`<EmptyState
message="No live data yet — pair a device.">`) until F4 fills it in
  with sensor cards.
- `apps/web/src/features/patients/tabs/PlaceholderTab.tsx` — new; one
  reusable placeholder used by Place / History / Alerts / Settings until
  their owning features ship.
- `apps/web/src/components/ui/tabs.tsx` — new if not present; shadcn
  Radix Tabs primitive.
- `apps/web/src/components/ui/badge.tsx` — new if not present; shadcn
  Badge primitive used by the connection-status pill.
- `apps/web/src/lib/usePatientStream.ts` — edit; widen the hook to
  return a status + lastSeen surface as well as accepting callbacks:
  ```ts
  export interface PatientStreamHandle {
    status: 'idle' | 'subscribed' | 'disconnected' | 'error';
    lastSeen: { sensor: number | null; position: number | null; alert: number | null };
  }
  export function usePatientStream(
    patientId: string | null,
    callbacks: PatientStreamCallbacks,
  ): PatientStreamHandle;
  ```
  Internally, the hook tracks `status` from the `subscribe` callback's
  status arg (`SUBSCRIBED` → `subscribed`, `CHANNEL_ERROR` → `error`,
  `CLOSED` / `TIMED_OUT` → `disconnected`). `lastSeen` is updated on
  every received row keyed by channel kind. The TODO comment at line 14
  about row schema imports is closed by F2/F4 work — this F3 edit
  doesn't depend on it.
- `apps/web/src/App.tsx` — edit; the `/patients/:id` route already
  ships in F2 wrapped around `PatientDetailPage` (F2 ships it pointing
  at a placeholder; F3 swaps the placeholder for the real component).
  No App.tsx change in F3 if F2 already routed it to
  `PatientDetailPage`; if F2 routed to a placeholder, F3 swaps the
  import.

## Risks

- **Hook signature widening is a breaking change for any earlier
  caller**. F0/F1 don't consume `usePatientStream` yet; F4 will. So the
  widening is safe in F3 — but if someone wires the hook into F2 in
  parallel, coordinate the rebase. Mitigation: keep the callback-only
  shape supported (return value is additive).
- **Pub/sub fanout vs. per-component subscription**. The context
  pattern adds indirection. Alternative: each component calls
  `usePatientStream` directly with its own callbacks, but they all
  point at the same `patient:<uuid>` channel — Supabase's client
  multiplexes channels of the same name to a single websocket
  subscription, so per-component subscribe might be free. **Decision**:
  the context is still preferred because it (a) centralises status +
  lastSeen, (b) gives F4's sparkline buffer a single owner regardless
  of cards rendering, (c) lets future tabs (F8 marker, F12 alert feed)
  consume without each understanding the channel shape. The pattern
  matches CROSS_CUTTING §7's recommendation that "live data lives in
  Zustand stores; React Query handles server state" — the context is
  effectively the Zustand-edge of that split.
- **Tab body remount**. If the Tabs primitive unmounts inactive tabs
  (most do, including Radix), in-tab component state is lost on
  switch — but the **stream context lives above the tabs** and so the
  channel survives. F4's sparkline buffer must therefore live in the
  Zustand store (per [CROSS_CUTTING §7](../CROSS_CUTTING.md#7-realtime-patterns)),
  not in component state. F4's doc captures this.
- **Deep-link to a tab the patient isn't ready for**. E.g.
  `?tab=place` before any floor plan exists. The Place tab itself
  handles the empty state when F5 ships; F3's placeholder works for
  Phase 1.
- **Patient header data churn**. Caregiver chips are derived from
  `caregiver_patient` joined to `caregivers`; this is a static-ish
  read, not realtime. If a peer caregiver self-removes (F1 policy
  permits), the chip is stale until refresh. Acceptable for V1.

- [ ] `npm run lint` clean.
- [ ] `npm run typecheck` clean.
- [ ] `npm run test` passes (the three new test files).
- [ ] `npm run build` clean.
- [ ] Manual smoke:
  1. `supabase start && npm run dev`. Sign in; click into a patient.
  2. Verify the header renders the name, age, and at least one
     caregiver chip.
  3. Click between Live / Place / History / Alerts / Settings tabs;
     confirm the URL updates and the connection pill stays "subscribed"
     across switches.
  4. Open the browser DevTools → Network → WS; observe a single
     `realtime` connection with one `patient:<uuid>` channel.
     Switching tabs does not open a new channel.
  5. Stop Supabase (`supabase stop`); verify the pill turns red
     within ~30 s and a friendly inline message renders.
  6. Restart Supabase; verify the pill returns to "subscribed" without
     a page reload.
  7. Visit `/patients/00000000-0000-0000-0000-000000000000`; verify
     the not-found empty state.
- [ ] Updates BACKLOG.md if anything was deferred (caregiver-chip
      realtime invalidation, mobile responsive header, cross-tab in-app
      notifications when the patient changes state).
