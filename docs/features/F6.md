# F6 — BLE Beacon Pairing & Placement

## Goal

A caregiver opens the Beacons panel inside the patient dashboard, watches BLE MAC addresses appear live as the wearable detects them, taps each beacon to label it ("Living room", "Hallway", "Bedroom"), and drags the labelled beacon to its real-world location on the F5 floor plan. On save, each beacon is a row in `beacons` with `floor_plan_id`, `(x_canvas, y_canvas)`, `mac_address`, and `label`. Three or more placed beacons unblocks F7 (calibration) and is the prerequisite for F8 indoor positioning.

## Acceptance criteria

From the build spec for F6 (BLE Beacon Pairing & Placement) and the Phase 2 exit criteria in [../PHASES.md](../PHASES.md#phase-2--place):

- The discovery view subscribes to live `signals` for the patient's device and renders a list of distinct BLE MACs as they arrive, with the most recent RSSI per MAC and a "first seen" timestamp.
- The caregiver can pair a discovered MAC by entering a label and confirming. Pairing inserts a row in `beacons` with `patient_id`, `mac_address`, `label`, and `floor_plan_id` (the patient's active floor plan).
- The caregiver can drag a paired but unplaced beacon onto the F5 canvas. The drop point sets `(x_canvas, y_canvas)`. Subsequent drags update the same row.
- A paired beacon can be unpaired (deleted), which cascades cleanly per the existing FK (`floor_plan_id ... on delete set null`; `patient_id ... on delete cascade`). The dashboard reflects the deletion within one realtime tick.
- At least 3 beacons can be paired and placed for one patient (Phase 2 exit gate); the UI surfaces "fewer than 3 beacons placed — F8 indoor positioning will not run" while count < 3, citing the F8 dependency.
- Empty / loading / error states present per [../CROSS_CUTTING.md §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states).

No spec corrections at the F6 level.

## Dependencies

**Hard prerequisites**

- F5 shipped — beacons need a canvas to be placed on. The patient must have a `floor_plans` row with a non-null `scale_meters_per_pixel`.
- `beacons` table from the foundation migration (already in place; see `supabase/migrations/20260504000000_init.sql` lines 80–94). The `tx_power` and `rssi_at_1m` columns are TODO-marked for F8 — F6 leaves them null on insert.
- RLS read policy on `beacons` (already in place). RLS **write** policies for `beacons` are added by this feature (see Migrations).
- `signals` realtime channel reaching the dashboard. The mqtt_bridge in Phase 1 must already be writing — but F6 needs `signals` to be **subscribable**. Two parts:
  1. The mqtt_bridge processes `device/{patient_id}/signals` per the schema in `packages/shared/src/mqtt/signals.ts`.
  2. The mock generator (Phase 2 entry criterion in [../PHASES.md](../PHASES.md#phase-2--place)) publishes signals.
  3. The dashboard's `usePatientStream` is extended to subscribe to a signals source. Since signals are deliberately not persisted (per [../PHASES.md Phase 3](../PHASES.md#phase-3--locate) discussion), the F6 discovery view cannot rely on a `signals` postgres-changes channel. **Decision**: the mqtt_bridge re-broadcasts validated signals on a Supabase Realtime broadcast channel `patient:<uuid>:signals` after persisting nothing. The dashboard subscribes via an extension to `usePatientStream` accepting an `onSignals` callback. See PARALLEL_TRACKS Phase 2 BE row.
- Cross-cutting decision [../CROSS_CUTTING.md §6 — Canvas state](../CROSS_CUTTING.md#6-canvas-state-pattern): beacons are addressable, separate rows in `beacons`, not embedded in `canvas_json`.
- Cross-cutting decision [../CROSS_CUTTING.md §7 — Realtime](../CROSS_CUTTING.md#7-realtime-patterns): one canonical hook `usePatientStream`. F6 extends, does not bypass.

**Soft prerequisites**

- Mock telemetry generator in `tools/mock-telemetry/` runs in `signals` mode publishing realistic BLE MACs at ~1 Hz (Phase 2 entry criterion).

**What this feature unblocks**

- F7 (calibration workflow) — calibration capture aggregates RSSI per **paired** beacon. Without F6 there are no beacons to key the signature against.
- F8 (indoor positioning) — needs `beacons` rows with `(x_canvas, y_canvas)`, `tx_power`, and `rssi_at_1m`. F6 writes the first two; F8 (or an inline calibration UX) writes the latter two.

## Contracts in `packages/shared`

F6 adds one published type, lives next to the MQTT schema since it shadows the wire shape:

- `packages/shared/src/mqtt/signals.ts` — already defines `SignalsMessage`, `BleSample`, `WifiSample`. F6 re-exports these for the dashboard's `onSignals` callback type. No new type needed; `SignalsMessage` is the contract.

The bridge → dashboard broadcast payload shape **is** the validated `SignalsMessage`. No stripping, no transformation. Both sides import from `@alzcare/shared`.

The shared `CalibrationSignature` type is owned by F7 ([./F7.md](./F7.md)); F6 does not produce it.

## Migrations

One migration in this feature.

- `<ts>_beacons_rls_writes.sql` — Adds insert / update / delete RLS policies on `public.beacons`. Pattern per [../CROSS_CUTTING.md §1](../CROSS_CUTTING.md#1-rls-write-policies): `for insert with check (public.is_caregiver_for(patient_id))`; same predicate for `using` on update / delete; `with check` mirrored on update. Rollback is `drop policy ... on public.beacons` per policy name; additive change, no data migration.

## RPCs / edge function changes

**Beacon pair / place** uses the standard RLS-gated insert / update path. The two columns the caregiver writes (`mac_address`, `label`, `(x_canvas, y_canvas)`, `floor_plan_id`) all sit on `beacons`; `is_caregiver_for(patient_id)` is sufficient guardrail. No SECURITY DEFINER function is needed and adding one would bypass the audit-log triggers we plan to attach in Phase 4 / 5.

**Trade-off considered**: a `pair_beacon(patient_id, mac, label)` RPC would let the bridge or a future onboarding flow create beacons under service role. F6 does not need this and adding it now invents an unowned API surface. Defer.

**The signals broadcast change is the one BE deliverable in F6**. It lives in `apps/edge/functions/mqtt_bridge/`. Strictly speaking this is a Phase 2 enhancement to the mqtt_bridge, not a new function. The bridge:

1. Continues to validate + persist telemetry / events as in Phase 1.
2. For `signals` payloads: validates against `SignalsMessage`, then calls `supabase.channel('patient:<patient_id>:signals').send({ type: 'broadcast', event: 'signals', payload: validated })`. No `signals` table write; the Phase 2 design deliberately does not persist raw signals (see [../PHASES.md](../PHASES.md#phase-2--place) and the F7 doc for the rationale).

This is recorded as a hand-off in [../PARALLEL_TRACKS.md Phase 2 BE row](../PARALLEL_TRACKS.md#phase-2--place).

- **Stale MAC pruning** (`useDiscoveredBeacons.test.ts`). A MAC absent from the stream for > 30 s renders as stale (yellow pip) per [../CROSS_CUTTING.md §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states); does not auto-remove (caregivers may want to pair a beacon they just walked past).
- **Pair flow** (`beaconQueries.test.tsx`). Click "Pair", enter "Living room", confirm; assert `from('beacons').insert({ patient_id, mac_address, label, floor_plan_id })` was called with the correct payload.
- **Placement flow** (`beaconQueries.test.tsx`). Drag a paired beacon to coordinates (450, 320); assert the update mutation fired with `{ x_canvas: 450, y_canvas: 320 }`.
- **Fewer-than-three guard** (`BeaconsPanel.test.tsx`). With 0, 1, 2 placed beacons, the "F8 indoor positioning will not run yet" notice renders. With 3, the notice is hidden.

## Risks

- **Signals broadcast vs. persistence.** The decision in [../PHASES.md Phase 2](../PHASES.md#phase-2--place) is to **not** persist raw signals — only computed `position_estimates` are stored. The dashboard therefore consumes signals via Realtime broadcast from the bridge, not via postgres_changes. If the broadcast channel proves unreliable in dev (e.g. ordering, dropped messages on reconnect), the fallback is a thin `signals` table with short retention. This trade-off is recorded in [../PHASES.md](../PHASES.md#phase-3--locate) under "If the project later needs replay-from-raw-signals". Do not add the table speculatively.
- **Minimum 3 beacons for F8.** Trilateration needs ≥ 3 non-colinear beacons; fingerprint matching tolerates fewer but degrades. Phase 2 exit gate is "place ≥ 3 beacons". UX must surface the count and the F8 dependency clearly. With < 3 placed beacons, the calibration UI from F7 still functions — it captures whatever signatures it sees — but the F8 marker will not run until the count reaches 3.
- **MAC duplicates across patients.** The schema has `unique (patient_id, mac_address)` — the same beacon can be paired to multiple patients (correct). Two simultaneously logged-in caregivers for the same patient pairing the same MAC will hit the unique violation; surface the error gracefully.
- **Beacon placement precision.** The caregiver eyeballs where a beacon physically sits on the canvas. F8's accuracy depends on this. Mitigation deferred to F8 (per [../PHASES.md Phase 2 Risks](../PHASES.md#phase-2--place)) — fingerprint matching is more forgiving than pure trilateration, and F7's calibration walk corrects for it.
- **Discovery noise.** A real environment surfaces dozens of unrelated BLE MACs (laptops, phones, neighbour devices). The discovery list grows long. Mitigation: sort by latest RSSI descending, optionally filter "show only > -85 dBm" toggle. Don't attempt deny-listing in V1.
- **Realtime channel teardown.** When the caregiver leaves the patient detail route, the `usePatientStream` hook's effect cleanup must `removeChannel` for both the postgres_changes channel and the new broadcast channel — failing to do so leaks subscriptions. Cover in the existing `usePatientStream` cleanup test.
- **Bridge's broadcast under service role.** The bridge already holds the service role key (per [../CROSS_CUTTING.md §1](../CROSS_CUTTING.md#1-rls-write-policies)). Realtime broadcasts using the service role bypass channel-level RLS — fine for V1 since channels are namespaced by `patient_id` and the dashboard only subscribes to channels for patients it has access to (caregivers can subscribe to any channel name; RLS-style channel auth is V2 / Supabase Realtime Authorization). Document this in `BACKLOG.md` under a new "Realtime broadcast channel auth" entry.
