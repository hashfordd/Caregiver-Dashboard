# F10 — Device Pairing

## Goal

A caregiver inside the patient detail dashboard pairs a wearable device to
the patient currently in context, sees the device's heartbeat (`last_seen_at`)
update as telemetry arrives, and can unpair the device when transferring or
retiring it. Closing F10 means the spine has a real `device_id` to anchor on:
F4's sensor cards display data from a known device, and the FK on
`sensor_readings.device_id` is satisfiable end-to-end. F10 is the only Phase
1 feature that mutates `devices`, and the policy surface it ships is the one
that prevents one caregiver from re-pairing another caregiver's wearable
mid-shift.

## Acceptance criteria

Restated from the spec and the Phase 1 critical path:

- A "Pair device" CTA lives in the Live tab of the patient detail
  dashboard. F3 reserves the slot; F10 implements it.
- The CTA opens a dialog that accepts a MAC address (and optionally a
  human label). On submit:
  - If a `devices` row with that MAC already exists and is unpaired
    (`paired_patient_id is null`), update it to point at the current
    patient.
  - If a row exists and is paired to a **different** patient, the
    submission is rejected with a clear error: "Device already paired
    to another patient. Unpair first." Spec: "prevent re-pairing a
    device already linked elsewhere."
  - If a row exists and is paired to the **current** patient already, no
    change; surface "Already paired."
  - If no row exists, insert a new `devices` row with the MAC,
    `paired_patient_id = <current>`, optional `firmware_version` from
    the form (or null).
- A device list panel within the Live tab shows all `devices` paired to
  the current patient, including:
  - MAC address
  - Optional label
  - `last_seen_at` formatted as relative time ("12 s ago", "2 min ago",
    "never").
  - An "Unpair" button (sets `paired_patient_id = null`; preserves the row).
- The `last_seen_at` column **updates without page reload** as the bridge
  writes telemetry. F10 does not write `last_seen_at` itself — the bridge
  does, on every persisted telemetry row. F10 reads it. The UI refreshes
  via either (a) realtime subscription on the `devices` table, or (b) a
  polling React Query refetch every 5 s. **Decision**: option (b) for V1.
  See [Risks](#risks).
- An RLS denial test confirms that caregiver A cannot pair a device by
  setting `paired_patient_id` to caregiver B's patient.

No spec corrections found for F10. The spec line "default device pairs to
the patient currently in context" is taken literally — there is no
free-floating device pairing screen; pairing always happens from inside a
patient detail page.

### Backend

- `supabase/migrations/<ts>_f10_device_write_surface.sql` — new; the
  device write policies and a partial unique index to enforce
  "MAC unique per pairing context":

  - `devices_allocated_insert` — `with check (paired_patient_id is not
null and is_caregiver_for(paired_patient_id))`. Inserting an
    unpaired device from the client is disallowed; pairing creates the
    row.
  - `devices_allocated_update` — `using (paired_patient_id is null or
is_caregiver_for(paired_patient_id))`
    `with check (paired_patient_id is null or
is_caregiver_for(paired_patient_id))`. The two-clause expression
    permits an allocated caregiver to set
    `paired_patient_id = null` (unpair), and to repaint a `null`
    device into their patient — but blocks a caregiver from rewriting
    a device paired to someone else's patient (the `using` clause
    evaluates the **pre-update** row).
  - `devices_allocated_delete` — `using
(is_caregiver_for(paired_patient_id))`. Hard delete is rare — V1
    surface uses unpair instead — but the policy is shipped for
    parity.

  Rollback note: drop policies (additive-no-rollback for table data).

- `supabase/migrations/<ts>_f10_pair_device_rpc.sql` — new; the
  SECURITY DEFINER RPC that handles the four-outcome flow atomically:

  ```sql
  create or replace function public.pair_device(
    p_mac_address text,
    p_patient_id uuid,
    p_label text default null
  ) returns public.devices
  language plpgsql security definer set search_path = public as $$
  declare
    caller uuid := auth.uid();
    existing public.devices;
    result public.devices;
  begin
    if caller is null then
      raise exception 'auth required';
    end if;
    if not public.is_caregiver_for(p_patient_id) then
      raise exception 'not allocated to patient';
    end if;
    select * into existing from public.devices where mac_address = p_mac_address;
    if existing.id is null then
      insert into public.devices (mac_address, paired_patient_id)
      values (p_mac_address, p_patient_id)
      returning * into result;
      return result;
    end if;
    if existing.paired_patient_id is not null
       and existing.paired_patient_id <> p_patient_id then
      raise exception 'device already paired to another patient'
        using errcode = 'P0001';
    end if;
    update public.devices set paired_patient_id = p_patient_id
      where id = existing.id
      returning * into result;
    return result;
  end$$;
  grant execute on function public.pair_device(text, uuid, text)
    to authenticated;
  ```

  **Why an RPC**: atomicity (the lookup-then-update window must be
  closed against concurrent pairing), uniform error-coding for the
  client, and centralised authorisation (the
  `is_caregiver_for(p_patient_id)` check is the single point of
  enforcement). Rollback note: drop function.

- `apps/edge/functions/mqtt_bridge/processMessage.ts` — edit (file
  shipped by F4); on every persisted telemetry row, also update

  the spec-mandated heartbeat. The bridge runs as service role; no

  `update`. **Coordination note**: F4 owns the file; F10 either lands
  in the same PR as F4's bridge work or amends it. Recommendation:

## Risks

- **`last_seen_at` realtime vs. polling**. Realtime on `devices` adds
  a publication entry and a subscription per patient detail page.
  Polling at 5 s is simpler, cheaper, and good-enough for V1. If
  caregivers need sub-5-s heartbeat freshness, switch to realtime in
  V2.
- **MAC collision across patients**. The current schema has
  `unique (patient_id, mac_address)` on `beacons`, but `devices` has
  global `unique mac_address`. That is correct: a physical device has
  one MAC, owned globally. The pair RPC respects this.
- **Concurrent pairing race**. Two caregivers, both allocated to
  different patients, both submit pair on the same unpaired MAC at
  the same moment. The RPC handles this: the first wins; the second
  reads the freshly-paired row and raises "paired to another
  patient". Verifiable via the RLS test harness if multi-session
  setup is in place.
- **RLS update policy `using` clause subtlety**. The two-clause
  expression `paired_patient_id is null or is_caregiver_for(...)`
  evaluates the **pre-update** row in `using` and the **post-update**
  row in `with check`. The semantics are: I can repaint a `null`
  device into my patient (pre is `null`, satisfies `using`; post is
  `is_caregiver_for`, satisfies `with check`); I can unpair my own
  device (pre is mine, satisfies `using`; post is `null`, satisfies
  `with check`); I cannot rewrite someone else's pairing (pre is
  not mine and not null, fails `using`). Confirm this with the RLS
  tests; they're the canary.
- **Heartbeat write contention**. Every telemetry row triggers a
  `devices` update. At 1 Hz × N patients, this is fine. At higher
  rates or N patients in the hundreds, the row-level lock on the
  device row could become contentious. Acceptable for prototype;
  document for V2.
- **Device label storage**. The current `devices` schema has no
  `label` column. F10's RPC accepts `p_label` but ignores it for now,
  OR F10 ships an additive column migration. **Decision**: ship the
  column. Add to the same `_f10_pair_device_rpc.sql` migration the
  ALTER:
  ```sql
  alter table public.devices add column if not exists label text;
  ```
  Or, more disciplined per
  [CROSS_CUTTING §12](../CROSS_CUTTING.md#12-migrations-discipline),
  a third migration `<ts>_f10_devices_label_column.sql`.
  Recommendation: third migration. Rollback: `alter table ... drop
column label;`.
