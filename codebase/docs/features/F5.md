# F5 — Indoor Floor Plan Editor

## Goal

A caregiver opens a patient's dashboard, picks "Floor plan", and draws their patient's space — outer walls, internal rooms, doors, key furniture — directly on a Fabric.js canvas. They tell the dashboard how long one drawn wall is in real-world metres, the canvas computes a `scale_meters_per_pixel` ratio, and the whole drawing is persisted. On reload the canvas reconstructs pixel-for-pixel from the saved state. The output is a saved `floor_plans` row that F6 (beacon placement) and F7 (calibration) consume as their canvas substrate.

## Acceptance criteria

From the build spec for F5 (Indoor Floor Plan Editor) and the Phase 2 exit criteria in [../PHASES.md](../PHASES.md#phase-2--place):

- The caregiver can draw walls, rooms, doors, and at least 6 furniture items on a Fabric.js canvas.
- The caregiver can set a real-world scale by selecting one drawn line and entering its length in metres; the dashboard computes `scale_meters_per_pixel` from the line's pixel length and persists it on `floor_plans`.
- A saved floor plan reloads pixel-for-pixel after a hard refresh — the canvas object count and geometry round-trip cleanly.
- Save / load round-trip writes `canvas_json` (decorative geometry) and `scale_meters_per_pixel` (numeric) to a single `floor_plans` row scoped to the active patient.
- A floor plan can be edited after creation. If `calibration_points` exist for the floor plan, the editor warns the caregiver on save that calibration data is now stale; it does not auto-invalidate (per [../CROSS_CUTTING.md §6](../CROSS_CUTTING.md#6-canvas-state-pattern)).
- Empty / loading / error states present per [../CROSS_CUTTING.md §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states).

No spec corrections at the F5 level. The Phase 2 acceptance text in [../PHASES.md](../PHASES.md#phase-2--place) captures it: "draws a 4-room floor plan with 6 furniture items, saves it, refreshes, sees it intact".

## Dependencies

**Hard prerequisites**

- Phase 1 closed: F1 auth, F2 roster, F3 dashboard shell. F5 is a tab inside the patient detail dashboard from F3.
- `floor_plans` table from the foundation migration (already in place; see `supabase/migrations/20260504000000_init.sql` lines 69–78).
- RLS read policy on `floor_plans` (already in place). RLS **write** policies for `floor_plans` are added by this feature (see Migrations).
- Cross-cutting decision [§6 — Canvas state pattern](../CROSS_CUTTING.md#6-canvas-state-pattern): addressable entities live in tables; decorative geometry lives in `floor_plans.canvas_json` JSONB. Restated below.

**Soft prerequisites**

- None. F5 does not consume realtime data; it is pure CRUD against `floor_plans`.

**What this feature unblocks**

- F6 (beacon placement) — beacons are dragged onto the F5 canvas and persisted with `(x_canvas, y_canvas)` referring to F5's coordinate system.
- F7 (calibration workflow) — calibration points are placed on the F5 canvas at known physical positions.
- F8 (indoor positioning) — the position marker is rendered over the saved floor plan in the same coordinate space.

## Contracts in `packages/shared`

F5 does not introduce any cross-package types — its data is local to the dashboard. The `floor_plans.canvas_json` blob is opaque Fabric output and should not be typed by `packages/shared` (Fabric owns the shape and bumping Fabric should not break a cross-package type contract).

The two cross-cutting decisions F5 leans on but does not own:

- `scale_meters_per_pixel` is a column on `floor_plans` (numeric). It is read by F6 (placement preview as metres) and by F8 (RSSI → distance modelling).
- `(x_canvas, y_canvas)` are pixel coordinates in the same Fabric coordinate system the editor renders in. F6 / F7 / F8 consume the same system. No transform layer.

## Migrations

One migration in this feature. Slug filename `<ts>_floor_plans_rls_writes.sql` per [../CROSS_CUTTING.md §12](../CROSS_CUTTING.md#12-migrations-discipline) (one logical change per migration).

- `<ts>_floor_plans_rls_writes.sql` — Adds insert / update / delete RLS policies on `public.floor_plans` per [../CROSS_CUTTING.md §1](../CROSS_CUTTING.md#1-rls-write-policies). The pattern: `for insert with check (public.is_caregiver_for(patient_id))`; same predicate for `using` on update and delete; `with check` mirrored on update. Rollback is `drop policy ... on public.floor_plans` per policy name; additive change, no data migration.

F6 and F7 each ship their own RLS-writes migration for their tables. F5's migration touches `floor_plans` only.

## RPCs / edge function changes

None. F5 uses the standard Supabase client `from('floor_plans').insert(...) / .update(...)` path, gated by the RLS policies added in this feature's migration. There is no behaviour the dashboard needs to perform server-side that exceeds what RLS-with-check covers — `patient_id` is set client-side and `is_caregiver_for(patient_id)` denies any caregiver writing for a patient they aren't allocated to.

A SECURITY DEFINER function would buy nothing here: the predicate is a simple allocation check, not a multi-row aggregation.

## Tests

Vitest, run via `npm run test --workspace apps/web`. Per [../CROSS_CUTTING.md §13](../CROSS_CUTTING.md#13-testing-strategy).

- **Canvas state round-trip** (`canvasState.test.ts`). Construct a Fabric canvas in jsdom with the Fabric Node entry, add 4 walls, 4 rooms, 6 furniture items at known coordinates and rotations, call `serializeCanvas`, then `deserializeCanvas` into a fresh canvas, assert object count equals input and per-object positions / rotations match within float tolerance. This is the primary correctness test for the canvas state pattern.
- **Scale computation** (`canvasState.test.ts`). Given a line of pixel length 600 and an input of 3 metres, `computeScale` returns `0.005` (m/px). Edge cases: zero metres rejected, negative metres rejected, non-finite numbers rejected.
- **Editor render flow** (`FloorPlanEditor.test.tsx`). Loading skeleton present while query pending; empty state shown when no floor plan exists; error state with retry on query failure; save invokes the upsert mutation with the serialised `canvas_json` and computed `scale_meters_per_pixel`.
- **Calibration-stale warning** (`FloorPlanEditor.test.tsx`). When `useCalibrationCount` returns > 0, save first opens the warning modal; "save anyway" proceeds, "cancel" does not call the mutation.

The canvas round-trip test is the Phase 2 verification gate item from [../PHASES.md](../PHASES.md#phase-2--place) ("Round-trip canvas state").

## Risks

- **Scale calibration mistakes**. If the caregiver picks the wrong line or mistypes metres, every downstream distance is off. Mitigation: the scale dialog displays "selected line: 600 px" and a live preview "scale will be 1 px = 0.005 m" before commit. Also: surface the current `scale_meters_per_pixel` on the editor at all times so a regression is visible. Re-running the scale flow after the fact replaces the previous value.
- **Canvas re-edits after calibrations are captured**. Per [../CROSS_CUTTING.md §6](../CROSS_CUTTING.md#6-canvas-state-pattern), the editor warns but does not auto-invalidate. The caregiver decides whether the change affects signal propagation. The warning explains the trade-off; documentation in `docs/features/F7.md` describes how to recapture if needed. **Restate**: do not delete `calibration_points` rows on a `floor_plans.canvas_json` update — the caregiver's prerogative.
- **Fabric.js bundle size**. Roughly ~100 KB gzipped per [../CROSS_CUTTING.md §16](../CROSS_CUTTING.md#16-performance-budgets). Mitigation: lazy-load the editor route via `React.lazy` + `Suspense` so the floor-plan code isn't in the initial bundle. The dashboard shell loads without Fabric.
- **Canvas accessibility**. Fabric canvases are not natively keyboard-navigable. Per [../CROSS_CUTTING.md §14](../CROSS_CUTTING.md#14-accessibility-floor), full canvas a11y is V2. F5 minimum: every toolbar action has a keyboard shortcut and a labelled button. Selection and movement via keyboard is deferred — note in the editor's empty state.
- **Multiple floor plans per patient**. The schema permits N rows per `patient_id`. F5 ships single-floor-plan-per-patient UX (the active plan is the most recent one). Note this in the editor empty state ("a patient currently has one floor plan; multi-floor support is V2"); track in `BACKLOG.md` if the user pushes back.
- **`canvas_json` size**. A typical home plan is < 50 KB. There is no enforced cap. If a future test surfaces a large plan, add a server-side check on insert (8 MB JSONB ceiling is the practical ceiling, well above realistic input).
