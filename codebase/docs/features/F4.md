# F4 — Live Sensor Visualisation

## Goal

The Live tab of the patient detail dashboard shows three sensor cards — heart
rate, SpO2, temperature — each with the latest reading, the recorded-at
timestamp, and a 5-minute sparkline of the metric. Each card carries a
freshness indicator: a green pip when the most recent reading is < 30 s old,
a yellow pip with a "Last updated 47 s ago" caption beyond that. Closing F4
means the **end-to-end live data path** of the spine works: a telemetry
message published by the mock generator (or firmware, when it ships) lands in
the database via the bridge, propagates through Supabase realtime, and renders
on the dashboard within 1 s p50 / 3 s p99 (TST-04 latency target).

## Acceptance criteria

Restated from the spec and the Phase 1 exit criteria:

- The Live tab renders three sensor cards: HR (bpm), SpO2 (%), Temperature
  (°C, displayed in the caregiver's preferred unit if F1 added the
  preference; defer °F for V1 — see Risks). Each card has a header label,
  a large numeric value, a unit, a "last updated" timestamp, and a
  sparkline.
- Each card subscribes to `PatientStreamContext` (from [F3.md](./F3.md))
  and updates within 1 s of a `sensor_readings` row arriving via
  realtime (TST-04 p50).
- The sparkline buffer covers the last 5 minutes per metric. Older points
  are evicted on every push.
- When no data has arrived for ≥ 30 s, the card's pip turns yellow and a
  "Last updated <N>s ago" caption appears. Threshold per
  [CROSS_CUTTING §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states).
- A connection-status pill in the patient header (provided by F3) reflects
  the realtime channel state. F4 does not own the pill, but the integration
  test (broker stop → red pill within 30 s) is part of the F4 smoke gate.
- TST-04 partial: median publish-to-render latency < 1 s; p99 < 3 s.
  Measured by the mock generator stamping each payload's `recorded_at` and
  comparing it to `Date.now()` at the card render. Add an instrumentation
  hook (logged in dev only) that records this; do not ship it in the
  production bundle.

The spec's "5 minutes of sparkline" is taken at face value. If data volume
becomes an issue (1 Hz × 5 min = 300 points per metric per card — trivial),
no compression is needed.

## Dependencies

**Hard prerequisites**

- [F3.md](./F3.md) — `PatientStreamContext` and the Live tab slot.
- [F10.md](./F10.md) — soft (see below) — F4 can develop against the mock
  generator without a real device row, but the end-to-end smoke needs a
  paired device.
- **Mock telemetry generator** at `tools/mock-telemetry/` —
  see [PHASES.md Phase 1 entry criteria](../PHASES.md#phase-1--spine).
  This is the BE/TST workstream's deliverable and is the only Phase 1
  dependency F4 cannot work around. Do not redesign the generator inside
  F4; consume it as given. If it does not exist when F4 starts, that's
  a Phase 1 entry-criteria failure, not an F4 risk.
- **`mqtt_bridge` promotion to long-running mode with shared
  `processMessage`** (BE-06) — see
  [PHASES.md Phase 1 critical path](../PHASES.md#phase-1--spine) and
  [CROSS_CUTTING §11](../CROSS_CUTTING.md#11-edge-function-runtimes-deno-vs-long-running).
  The bridge must validate **and persist** to `sensor_readings` for the
  realtime channel to fire. Without this, F4's realtime subscription
  receives nothing. This is BE work; F4 does not implement it but cannot
  pass acceptance without it.
- [CROSS_CUTTING §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states) —
  loading skeleton, empty state, error retry, stale pip.
- [CROSS_CUTTING §7](../CROSS_CUTTING.md#7-realtime-patterns) —
  sparkline buffer goes in a Zustand store (live data, not server
  state).

**Soft prerequisites**

- F10 — F4 can develop against any `device_id` the mock generator emits.
  Until F10 ships, the device row may not exist and the FK on
  `sensor_readings.device_id` will reject inserts. Two paths:
  (a) seed a fixture device in dev; (b) make the mock generator
  insert/upsert a device row before publishing. Recommendation: (b),
  documented in the mock generator README.

**What this feature unblocks**

- The Phase 1 verification gate. F4 is the closing feature of the spine:
  passing TST-04 + the integration tests in
  [PHASES.md Phase 1 integration tests](../PHASES.md#phase-1--spine)
  closes Phase 1.

## Files to create or modify

### Frontend

- `apps/web/src/features/patients/tabs/LiveTab.tsx` — edit (F3 lands the
  placeholder); replace the placeholder with a flex grid of three
  `<SensorCard>` components and a `<DevicePairingSlot>` placeholder
  filled in by F10.
- `apps/web/src/features/patients/live/SensorCard.tsx` — new; props
  `metric: 'hr' | 'spo2' | 'temp'`. Reads from the live store
  (`useLiveSensorStore(s => s.cards[patientId][metric])`). Renders a
  Card with header (metric label + freshness pip), numeric value
  (large), unit, "Last updated Xs ago", and a `<Sparkline>` SVG.
  Handles the four states from CROSS_CUTTING §4.
- `apps/web/src/features/patients/live/Sparkline.tsx` — new; pure-SVG
  sparkline (no Recharts dep). Takes `points: { t: number; v: number
}[]` and renders a polyline plus min/max ticks. Pure SVG keeps the
  bundle small and avoids the Recharts pin until F13.
  **Why not Recharts**: see Risks. The decision is to defer the
  Recharts install to F13 where the history charts genuinely need axis
  formatting, tooltips, and zoom — features the F4 sparkline does not
  need.
- `apps/web/src/lib/stores/liveSensorStore.ts` — new; Zustand store
  per [CROSS_CUTTING §7](../CROSS_CUTTING.md#7-realtime-patterns):
  ```ts
  type Metric = 'hr' | 'spo2' | 'temp';
  interface CardState {
    latest: { value: number; recordedAt: number } | null;
    buffer: { t: number; v: number }[]; // 5 min ring
    lastReceivedAt: number | null;
  }
  interface LiveSensorState {
    cards: Record<string, Record<Metric, CardState>>; // keyed by patientId
    pushReading: (patientId: string, row: SensorReadingRow) => void;
    reset: (patientId: string) => void;
  }
  ```
  `pushReading` decomposes the row into three metric updates,
  appends to each buffer, evicts entries older than 5 min, updates
  `lastReceivedAt`. The store is the single owner of the sparkline
  state across tab remounts (Live tab unmounts and remounts as the
  caregiver switches tabs, but the store survives).
- `apps/web/src/features/patients/PatientStreamContext.tsx` — edit
  (this is F3's file but F4 wires its consumer); inside the provider's
  `usePatientStream` callback, dispatch each sensor reading to
  `useLiveSensorStore.getState().pushReading(patientId, row)`. Also
  invoke the registered subscriber callbacks (the existing F3 fanout).
  This double-write is intentional: components that want streaming
  callbacks (e.g. F12 alert flash) keep using the fanout; components
  that want a state slice (sensor cards) read the store. The store
  write happens **at the provider level**, not inside the card, so a
  card unmount/remount does not lose buffer.
- `apps/web/src/lib/time.ts` — new (see
  [CROSS_CUTTING §5](../CROSS_CUTTING.md#5-time-handling)); exports
  `formatTimestamp(iso, format)` and `secondsAgo(iso)`. F4 is the
  first consumer; F12/F13 will reuse.
- `apps/web/src/lib/stores/index.ts` — new; barrel re-export for
  Zustand stores. Future stores (alert feed, position marker) live
  alongside.

  long-running entry points. Inputs: `(topic: string, message: unknown,

## Risks

- **Recharts not installed yet** — F4's sparkline is intentionally
  hand-rolled SVG, so this risk is closed for F4. Recharts lands when
  F13 (history charts) needs it. Pin the version at F13 install time;
  do **not** install eagerly. Add to BACKLOG removal: when Recharts
  finally lands, remove the BACKLOG entry "Front-end libs not yet
  installed → Recharts".
- **Mock generator schema drift**. The Zod schemas in
  `packages/shared/src/mqtt/` are the SSOT. Mock generator imports
  them directly; if BE changes a schema, the generator's TS check
  breaks before runtime.
- **Stale-data threshold tuning**. 30 s is the
  [CROSS_CUTTING §4](../CROSS_CUTTING.md#4-loading--empty--error--stale-states)
  default. Real firmware may publish at 0.5 Hz (every 2 s) or batch.
  If the realistic interval is closer to 30 s, the threshold is too
  aggressive. Mitigation: parameterise the threshold per metric in the
  store; tune in Phase 5 once firmware is real. Document the default
  in the store.
- **Latency target ambiguity**. 1 s p50 is achievable on a local
  stack; production with hosted Supabase + a Fly.io bridge needs a
  measured run. Mitigation: ship the dev-only instrumentation logger
  in F4; remove for prod. The TST-04 number is a Phase 1 verification
  gate, not an F4 acceptance — F4 passes when the path works; the
  number is measured by TST.
- **Sparkline rendering perf**. SVG polyline with 300 points re-renders
  on every tick. React 18's automatic batching helps. If perf
  degrades, switch to `<canvas>` (still no Recharts). Acceptable for
  V1.
- **Unit display (°C/°F)**. Spec says caregiver preference. F1's
  profile shipped without the unit toggle (out of F1 scope). F4
  defaults to °C; the toggle is a follow-up. Add to BACKLOG.
- **F10 race**. If a device-pair-and-publish flow races (caregiver
  pairs while the mock is publishing under a stub `device_id`),
  inserts may fail FK. Acceptable for V1; mitigate by always running
  the mock against the configured paired device.

## Definition of done

- [ ] All [acceptance criteria](#acceptance-criteria) met.
- [ ] `npm run lint` clean.
- [ ] `npm run typecheck` clean.
- [ ] `npm run test` passes (the four new test files + the appended RLS
      case + the bridge test).
- [ ] `npm run build` clean.
- [ ] Manual smoke (the Phase 1 verification rehearsal):
  1. `supabase start && npm run broker:up && npm run dev`. Confirm the
     bridge container is running.
  2. Sign in; pair a device to the test patient (F10) — or, before
     F10 lands, run a one-off SQL `update devices set
paired_patient_id = '<id>' where mac_address = '<mac>'`.
  3. Start the mock generator with the patient_id and device_id of
     the paired device, publishing telemetry at 1 Hz.
  4. Open the Live tab. Within 1 s, all three cards begin updating
     with values from the generator.
  5. Stop the generator. Within 30 s, all three cards' pips turn
     yellow and the captions show the seconds-since-last-reading.
  6. Restart the generator. Within 1 s, the pips return to green and
     values resume.
  7. Stop the broker (`npm run broker:down`). Within 30 s, the
     header connection pill turns red. Restart the broker; the pill
     returns to green without a page reload.
  8. From caregiver B's session, attempt to read caregiver A's
     `sensor_readings` via the SQL editor; confirm zero rows.
- [ ] Updates BACKLOG.md if anything was deferred (°F unit toggle,
      Recharts removal-on-install bookkeeping, latency instrumentation
      kept past Phase 1, stale-threshold tuning per metric).
