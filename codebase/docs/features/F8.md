# F8 — Indoor Position Triangulation Engine

## Goal

When the wearable publishes a `signals` payload (BLE + WiFi RSSI samples, optionally GPS) at ~1 Hz, the system computes a smoothed indoor position estimate, writes a `position_estimates` row, and the dashboard renders an animated marker on the floor plan whose opacity tracks confidence. The estimate is the fused output of trilateration over the strongest BLE beacons and a kNN fingerprint match against captured `calibration_points`, then smoothed against the last 5 estimates for the patient. Indoor accuracy target: < 1.5 m on 80% of samples in calibrated tests.

This is the single hardest feature in the build. It is the verification gate to Phase 4 — Phase 4's zone-rule firing (F11) consumes `position_estimates` and is meaningless if the estimates aren't trustworthy.

## Acceptance criteria

- Position marker on the floor plan updates at ≥ 1 Hz with smoothed, non-flickering motion.
- Indoor position error < 1.5 m on 80% of samples in calibrated tests (TST-05).
- Marker opacity is bound to `position_estimates.confidence` (0..1).
- Indoor↔outdoor mode switch is hysteretic: no mode flip within 5 s of the previous flip (POS-08).
- Beacon dropout (one beacon to -127 dB mid-sequence) does not break positioning; fingerprint matcher recovers within 2 estimates.
- NLOS spike (one beacon +20 dB) is dampened by smoothing; no jump > 2 m in a single tick.
- The pipeline is implemented as pure functions in `packages/shared/src/positioning/` and orchestrated by `apps/edge/functions/position_estimator/`. No algorithm code in the edge function or in the web app.

The indoor accuracy criterion is what F7 originally claimed; per [PHASES.md Phase 2 Risks](../PHASES.md#phase-2--place), it actually belongs here. Phase 2's F7 acceptance is calibration data quality only.

## Algorithm pipeline

One `SignalsMessage` payload → one `position_estimates` row. The pipeline lives in `packages/shared/src/positioning/` as composable pure functions; `position_estimator` orchestrates them. Walk the stages:

### 1. RSSI → distance (POS-01, POS-02)

Log-distance path-loss model, per beacon:

```
distance_m = 10 ^ ((rssi_at_1m - rssi_observed) / (10 * pathLossExponent))
```

- `rssi_at_1m` and `tx_power` come from the `beacons` row (columns added in foundation migration; populated during F6 beacon pairing).
- Path-loss exponent default is 2.0 (free space). Tunable per floor plan via a future `floor_plans.path_loss_exponent` column (deferred — see Risks). For now, accept it as a parameter with default 2.0.
- Output: `{ beacon_id, distance_m, rssi }[]` — one entry per beacon seen in the payload that has a calibrated row.
- Beacons not in the `beacons` table for this patient are dropped (not used). Beacons with null `rssi_at_1m` are dropped with a warning log.

Pure function: `pathLossDistance(rssi, rssi1m, exponent) → number`. Vector wrapper: `rssiVectorToDistances(samples, beacons, exponent) → BeaconDistance[]`.

### 2. Trilateration (POS-03)

Take the **top 3 BLE beacons by RSSI** (strongest signals; lowest distance error). Solve for `(x, y)` in canvas pixel coordinates.

- Wrap `trilateration.js` (npm `trilateration` 1.0.4) for the solver, OR implement directly. Recommendation: implement directly. The math is small (least-squares over three circles), the library is unmaintained, and a hand-rolled solver lets the function return both the estimate and the **residual** (RMS distance error of the estimate against the three measured distances), which the fusion stage needs for confidence weighting.
- Reject solutions where:
  - Fewer than 3 calibrated beacons present in the payload → return `null` (fall back to fingerprint-only).
  - The 3 beacons are colinear (degeneracy check via the area of the triangle formed by their canvas coordinates < epsilon).
  - Residual > 5 m (in canvas units converted via `scale_meters_per_pixel`) — geometrically inconsistent.
- Output: `{ x_canvas, y_canvas, residual_m } | null`.

Pure function: `trilaterate(beaconDistances, beacons, scaleMetersPerPixel) → TrilaterationResult | null`.

### 3. Fingerprint match (POS-04)

Compose the current observation as a `Signature` (BLE entries `{mac, rssi}` + WiFi entries `{bssid, rssi}`). For every `calibration_points` row attached to the patient's active `floor_plan_id`, compute the Euclidean distance in RSSI space between the live signature and the calibration's stored signature.

- Missing entries on either side are penalised: treat absent BLE/WiFi as RSSI = -100 dB (a reasonable noise floor) for the distance calculation. This is critical — a calibration point that has 5 beacons but the live observation only has 3 must not match perfectly.
- kNN with k = 3. Weighted average of the top-k canvas coordinates by inverse distance: `weight_i = 1 / (rssi_distance_i + epsilon)`, then `position = Σ(weight_i * pos_i) / Σ(weight_i)`.
- Confidence: inversely proportional to the k-th neighbour's RSSI distance. A perfect match → high confidence; the nearest neighbour 30 dB away → low.
- Output: `{ x_canvas, y_canvas, k_distance } | null` (null if no calibration points exist).

Pure function: `fingerprintMatch(observation, calibrationPoints, k) → FingerprintResult | null`.

### 4. Fusion (POS-05)

Weighted blend of trilateration + fingerprint:

```
trilat_confidence = 1 / (1 + residual_m)         // 0..1, higher when residual low
fingerprint_confidence = 1 / (1 + k_distance/20) // 0..1, higher when k-distance low
weight_t = trilat_confidence
weight_f = fingerprint_confidence
fused.x = (weight_t * trilat.x + weight_f * fp.x) / (weight_t + weight_f)
fused.y = (weight_t * trilat.y + weight_f * fp.y) / (weight_t + weight_f)
```

- If trilateration returned null, use fingerprint alone (and vice versa).
- If both returned null, return null — the caller writes no row, and the realtime stream stays at the last known position.
- Output: `{ x_canvas, y_canvas, fused_confidence } | null`.

Pure function: `fuse(trilatResult, fingerprintResult) → FusedResult | null`.

### 5. Smoothing (POS-06)

Stateless edge function — the function does not retain history between invocations. Smoothing state lives in the database. The orchestrator queries the last 5 `position_estimates` rows for the patient (descending `recorded_at`), then applies a weighted moving average:

```
smoothed.x = Σ(w_i * row_i.x) / Σ(w_i)   for i in [latest, ..., latest-4, current]
weights = exponential decay, e.g. [0.5, 0.25, 0.125, 0.075, 0.05] for the past 5
                          + 1.0 for the current fused result
                          (re-normalised)
```

- 5 rows is enough to dampen single-tick spikes without lagging real motion meaningfully at 1 Hz.
- Why stateless: prototype scale is one patient at ~1 Hz. The extra DB read costs a few ms; not worth the complexity of a stateful service. [PHASES.md Phase 3 Recommendation](../PHASES.md#phase-3--locate) explicitly endorses this.
- If fewer than 5 rows exist (cold start), use what's available; the moving average degrades gracefully.

Pure function: `smooth(currentFused, recentEstimates) → SmoothedResult`. The `recentEstimates` argument is supplied by the orchestrator from the DB.

### 6. Confidence scoring (POS-07)

Combine three components into a 0..1 scalar:

```
signal_availability = min(1, beaconCount / 3)        // saturate at 3 beacons
match_quality       = fused_confidence (from step 4) // 0..1
smoothness          = 1 / (1 + jump_m)               // jump_m = distance from previous estimate
confidence = 0.4 * signal_availability + 0.4 * match_quality + 0.2 * smoothness
```

Surfaced in the UI as marker opacity (clamped to a minimum visible value, e.g. 0.3, so the marker never disappears).

Pure function: `scoreConfidence(inputs) → number` clamped to [0, 1].

### 7. Mode decision (POS-08)

Independent of indoor positioning: decide whether this estimate's `mode` is `'indoor'` or `'outdoor'`. Hysteretic — see [Mode-switch hysteresis](#mode-switch-hysteresis-pos-08) below.

### 8. Insert

Write a `position_estimates` row:

```ts
{
  patient_id,
  recorded_at: signals.recorded_at,    // preserve the wearable's clock
  mode: 'indoor' | 'outdoor',
  x_canvas: smoothed.x,                // null if mode === 'outdoor'
  y_canvas: smoothed.y,                // null if mode === 'outdoor'
  lat: gps?.lat ?? null,               // populated if mode === 'outdoor'
  lng: gps?.lng ?? null,               // populated if mode === 'outdoor'
  confidence,
}
```

Realtime publication is already configured in the foundation migration — the row appears on the `patient:<uuid>` channel within a few hundred ms.

### Why this pipeline order

- Trilateration is geometrically meaningful but noisy under multipath and NLOS.
- Fingerprint is empirically calibrated to **this** space and forgiving of NLOS — but useless if the layout drifts.
- Blending both buys robustness against either's failure mode.
- Smoothing only at the end so genuine motion isn't suppressed by upstream stages.

This rationale is from [PHASES.md Phase 3](../PHASES.md#phase-3--locate). Restated here so the F8 doc stands alone.

## Mode-switch hysteresis (POS-08)

Hysteresis prevents the view flapping between indoor and outdoor near doorways. Rules:

- **Switch to outdoor** when, for ≥ 5 consecutive seconds:
  - GPS confidence > 0.7 (HDOP < 2.0 and `fix_age_s < 5`), AND
  - Indoor confidence (from step 6 above) < 0.3.
- **Switch to indoor** only when GPS has been lost (no fix or `fix_age_s > 5`) for ≥ 5 s.

State source: query the last N `position_estimates` rows for the patient (where N covers ≥ 5 s of history at the publishing rate, so ~6 rows at 1 Hz). Count consecutive rows matching the switch condition. The orchestrator already fetches recent rows for smoothing — extend the same query to cover the hysteresis window.

Why this approach: zero extra schema, no in-memory state. [PHASES.md Phase 3](../PHASES.md#phase-3--locate) recommends this option (a) over a dedicated state column (b) for prototype simplicity.

## Trigger model

How `position_estimator` gets invoked. **Decision (per [PHASES.md Phase 3](../PHASES.md#phase-3--locate))**: the long-running `mqtt_bridge` invokes `position_estimator` via internal HTTP **after** validating a signals payload. The bridge does not persist signals; raw RSSI vectors are computationally cheap to ingest and storage-expensive at volume.

- The bridge calls `POST {SUPABASE_URL}/functions/v1/position_estimator` with the validated `SignalsMessage` body and `Authorization: Bearer ${SUPABASE_SERVICE_ROLE_KEY}`.
- `supabase/config.toml` currently has `verify_jwt = true` for `position_estimator`. Service-role calls bypass the JWT check (the service role is a privileged credential, not a user JWT). This is fine; document it inline in the function.
- Alternative considered: database webhook on `signals` row insert. Rejected — would require persisting signals, contradicting the storage decision.
- Alternative considered: queue-and-worker. Rejected — overkill for prototype scale (~1 Hz, one patient).

If, in V2, the project needs replay-from-raw-signals for tuning POS-10, persist signals to a dedicated `signals` table and have the bridge write there in addition to invoking the estimator. Defer until needed.

## RPCs / edge function changes

### `position_estimator` (real implementation)

- **Trigger**: HTTP POST from the bridge.
- **Auth**: service role only. The function rejects requests where `Authorization` does not match `Bearer ${SUPABASE_SERVICE_ROLE_KEY}`. `verify_jwt = true` in `supabase/config.toml` for this function is preserved — service-role calls bypass the JWT verification by design (Supabase treats the service role key as equivalent to a verified privileged JWT).
- **Request body**: a parsed `SignalsMessage` (the bridge has already validated it; the function re-validates as defence in depth).

  ```ts
  // input
  { v: 1, patient_id, device_id, recorded_at, ble: [...], wifi: [...], gps?: {...} }
  ```

- **Response**:

  ```ts
  // 200 OK
  { ok: true, mode: 'indoor' | 'outdoor', confidence: number, position_estimate_id: string }
  // 200 OK with no insert (insufficient data)
  { ok: true, skipped: true, reason: 'no_calibration' | 'no_beacons' | 'no_signal' }
  // 4xx
  { ok: false, error: string, issues?: ZodIssue[] }
  ```

- **Side effects**: one `position_estimates` row insert (or none, if the pipeline returned null).

### `mqtt_bridge` (extend)

- After validating a `signals`-topic payload, fire-and-await the POST to `position_estimator`. Don't block telemetry processing on signals — they're independent code paths.

No new RPCs; no `acknowledge_alert`-style stored functions.

## Tests

F8 has the heaviest test surface in the build. Organised by layer:

### Pure-function unit tests (Vitest, in `packages/shared`)

Located in `packages/shared/src/positioning/__tests__/`. Required:

- **pathLoss.test.ts** — round-trip a known distance through the formula at exponents 1.8, 2.0, 2.5; assert RSSI → distance and the inverse agree within numerical tolerance. Edge: rssi == rssi_at_1m → distance == 1 m.
- **trilaterate.test.ts** — three beacons forming an equilateral triangle of side 4 m; place a "patient" at the centroid; assert recovered position within 0.1 m of truth. Add 1 dB Gaussian noise to each input distance; assert recovery within 0.5 m. Test colinear-beacon rejection: three beacons on a line returns null. Test fewer-than-3 beacons returns null.
- **fingerprint.test.ts** — synthetic calibration set of 8 points in a 4×3 m space, each with 4 BLE entries; perfect-match observation returns the exact point's coordinates. Observation matching one point well and others poorly returns near that point. Empty calibration set returns null. Missing-entry penalty: an observation with one fewer beacon than the calibrations should still match the geometrically-closest point, not a random one.
- **fuse.test.ts** — trilat null + fingerprint present → fingerprint output; trilat present + fingerprint null → trilat output; both present with high trilat confidence → output near trilat; both null → null.
- **smooth.test.ts** — stationary history (5 identical points) + a +5 m spike → smoothed output is < 1 m from history (proves dampening). Linear motion (history of constantly-moving points) + consistent next point → smoothed tracks the motion (proves it doesn't over-dampen).
- **mode.test.ts** — sequence input. (a) 4 consecutive (gps_strong + indoor_weak) candidates → still indoor; the 5th → outdoor. (b) After mode flip, 4 (gps_lost) candidates → still outdoor; the 5th → indoor. Edge: alternating patterns never flip.
- **confidence.test.ts** — three beacons + perfect match + zero jump → near 1.0. Zero beacons → near 0.0. Perfect match but 5 m jump → mid-range (smoothness penalty applies).
- **pipeline.test.ts** — end-to-end on a fixture: synthetic signals + beacons + calibrations → expected output within tolerance. Locks the public contract.

### Edge function HTTP test (Vitest in Node mode)

`apps/edge/functions/position_estimator/__tests__/index.test.ts` — drive the function with a seeded local Supabase. Cases:

- Valid signals with beacons + calibration → row written.
- Missing service-role auth → 401.
- Invalid signals body → 400 with Zod issues.
- Patient with no beacons → 200 + skipped + reason.
- Patient with no calibration → 200; trilateration-only path runs.

### POS-10 synthetic RSSI replay harness

Lives at `tools/replay-signals/` (new directory). A Node CLI that:

1. Reads a fixture file (`fixtures/walk-1.jsonl`) of pre-recorded `SignalsMessage` payloads. Each line is one payload, in publish order, with timestamps.
2. POSTs each payload to the bridge's HTTP entry (or directly to `position_estimator` for tightest-loop testing) at the recorded interval (default) or at a configurable rate (`--rate 10x`).
3. Logs each resulting `position_estimates` row.
4. Optionally takes a `--truth fixtures/walk-1-truth.jsonl` of ground-truth canvas coordinates per timestamp; emits an **accuracy report** (mean / 80th-percentile / max error in metres).

Ownership: TST track. Drives both the unit-style accuracy harness (run against the pipeline directly) and the full-chain integration test (run through the bridge). Must produce a reproducible accuracy number for the F8 verification gate.

### Beacon dropout test

A fixture variant `fixtures/walk-1-dropout.jsonl` where one beacon's RSSI drops to -127 at sample index 50. Replay; assert the position-estimate stream:

- Does not jump > 2 m at the dropout instant (smoothing).
- Is back within accuracy budget within 2 estimates (fingerprint matcher takes over).

### NLOS spike test

A fixture variant where one beacon's RSSI spikes +20 dB for 3 consecutive samples (simulated reflection). Replay; assert no estimate jumps > 2 m in a single tick.

### Mode-switch hysteresis test

Driven by replay or by the `mode.test.ts` unit test. End-to-end variant: replay a fixture that toggles GPS confidence on/off rapidly; assert no two mode flips occur within 5 s of each other across a 60 s window.

### Frontend marker test

`PatientMarker.test.tsx` — RTL: render with three successive `PositionEstimateRow` snapshots dispatched through a mocked `usePatientStream`. Assert:

- Opacity tracks confidence (clamped to ≥ 0.3).
- Position updates between renders.
- Component cleans up Fabric event handlers on unmount.

## Risks

- **Accuracy target may not be met**. 1.5 m / 80% with cheap BLE beacons in an environment with multipath, NLOS, and movement is aggressive. Mitigations from [PHASES.md Phase 3](../PHASES.md#phase-3--locate):

  - Dense calibration in F7 (8+ points in a 4-room space — exit criterion).
  - Per-beacon `tx_power` and `rssi_at_1m` populated during F6 pairing (not assumed defaults).
  - Lean on fingerprint match more heavily than pure trilateration via fusion weights.
  - If still missing, **report achieved accuracy honestly in EV-05**. The dashboard is not graded on hitting an arbitrary number; the evaluation report is graded on rigour of measurement and explanation. Don't tune the test to pass; tune the algorithm and document the result.

- **Stateless smoothing limitation**. Reading the last 5 rows per invocation adds a DB round-trip (~5–20 ms typical). At 1 Hz × one patient that's negligible. At 10 Hz × 50 patients it would matter. V2 should move to a stateful in-memory service. Documented in [PHASES.md Phase 3 Risks](../PHASES.md#phase-3--locate).

- **Signal availability in cold start**. The first estimate has no history to smooth against. Pipeline degrades gracefully (smoothing returns the unmodified fused result with `jump_m = 0`). Confidence on the first sample will be lower than steady-state — that's correct, the system is honest about uncertainty.

- **Floor-plan changes invalidate calibrations**. If the caregiver edits the canvas after calibrating, the canvas coordinates of the calibration points still refer to the old layout. F7 / [PHASES.md Phase 2 Risks](../PHASES.md#phase-2--place) handles this by warning on save, not auto-deleting. F8's behaviour: still uses the calibrations (the matcher is robust to a few stale points), but the accuracy report should warn if calibrations are older than the floor plan's `updated_at`. Add this check to the orchestrator and surface as a UI warning.

- **Trilateration solver hand-rolled vs library**. Hand-rolled risks subtle bugs. Mitigation: dense unit tests with known geometry (equilateral triangle, right triangle, scaled, perturbed). Library (`trilateration` npm) risks unmaintained ESM/Deno compat. If hand-rolling proves time-expensive, fall back to the library and pin exact: `"trilateration": "1.0.4"`.

- **`verify_jwt = true` + service-role-only**. The function is callable only by the bridge. If the bridge URL or service-role key leaks, an attacker could DoS the function. Mitigation: keep the service-role key in server-side env only ([CROSS_CUTTING §1](../CROSS_CUTTING.md#1-rls-write-policies)). Don't add a public endpoint.

- **Fabric.js marker performance**. Updating canvas at 1 Hz is fine; stacking up un-disposed Fabric objects is not. The `PatientMarker` must reuse a single Fabric circle and update its `set({ left, top, opacity })` per render — not create-and-destroy.
