# F12 — Alert Feed & Acknowledgement

## Goal

A caregiver sees a header bell that surfaces new alerts within 2 seconds of their being written. Opening the bell shows recent unacked alerts across all allocated patients. Each patient's dashboard has an **Alerts** tab listing the patient's own alert history. Acknowledgement is a single click, persists across reload, and propagates to other open tabs of the same caregiver within 2 seconds. Critical alerts produce an audible cue when the tab is active and a desktop notification when the tab is inactive.

## Acceptance criteria

- An alert row inserted into `alerts` for a patient the caregiver is allocated to appears in the bell's unacked list within **2 s** of insert (realtime path; not poll).
- The bell badge counts unacked alerts across allocated patients and visually distinguishes presence of any `critical` (red dot vs. neutral pip).
- The per-patient **Alerts** tab lists the patient's alerts, severity-tagged and ack-state-tagged, in reverse-chronological order. Filter chips: severity (`info` / `warn` / `critical`), state (`active` / `acknowledged`).
- The **Acknowledge** button calls the `acknowledge_alert` RPC (defined in [F11](./F11.md)). The RPC is idempotent. On success, the row's `acknowledged_at` and `ack_by_caregiver_id` populate.
- Acknowledgement persists across reload (the row's columns are persisted server-side, not local state).
- Acknowledgement in tab A propagates to tab B (same caregiver) within 2 s via the `alerts` realtime channel — tab B sees the row leave the active list.
- **Critical** alerts produce an audible cue (browser tab audio) when the tab is active. When the tab is inactive (`document.hidden`), a `Notification` appears via the Notification API. Permission is requested gracefully; denial is non-fatal.
- Severity colour map per [CROSS_CUTTING.md §15](../CROSS_CUTTING.md#15-brand-consistency): `info` = steel, `warn` = orange, `critical` = the defined red token. The red token is added to `globals.css` as part of this feature.

## Contracts

F12 has no shared package surface of its own — it consumes the `AlertRule`, `AlertSeverity`, and `Alert` row shape from F11 / the migration. The new types it owns are local to the web app:

- `AlertCueLevel = 'silent' | 'audible' | 'audible+notify'` — a small derived type for the cue policy. The current implementation is hard-coded: `info | warn = silent`; `critical = audible+notify`. Future tunability sits behind a setting; out of scope for V1.

## Migrations

F12 does not introduce schema changes. The `acknowledge_alert` RPC ships with [F11](./F11.md). The `alerts` realtime publication entry was added in the foundation migration.

If during implementation it becomes clear a column is needed (e.g. `last_seen_at` on `alerts` for "I saw it but haven't acked"), open a separate migration under [CROSS_CUTTING.md §12](../CROSS_CUTTING.md#12-migrations-discipline)'s one-logical-change-per-file rule. Not in scope for V1.

## RPCs / edge function changes

- **`acknowledge_alert(alert_id uuid)`** — defined in [F11](./F11.md). F12 calls it from `AckButton`. The client-side wrapper:

```ts
const { data, error } = await supabase.rpc('acknowledge_alert', { p_alert_id: alertId });
```

On error: surface a toast, do not roll back the optimistic update unless the error is anything other than the idempotent "already acked" path.

No edge function changes in F12.

## Tests

- **TST-09 alert delivery reliability** — covered jointly with [F11](./F11.md). F12's responsibility within TST-09: assert each delivered alert renders in the bell within 2 s, regardless of which patient it belongs to.
- **Ack persists across reload** — write an alert, ack it via the RPC, reload the page, assert the row still renders as acknowledged with the correct timestamp.
- **Ack propagation in tab A → tab B** — the cross-tab integration test above. The simplest reliable form of this test mocks the realtime client; the manual smoke is two browser tabs of the same caregiver session.
- **Critical cue tests** — see file list. The audio playback itself can't be asserted in jsdom; assert the `AudioContext` was created and an oscillator was started. The notification path asserts `Notification` was constructed when and only when `document.hidden`.
- **Filter behaviour** — set the filter to `state=acknowledged`; assert the active items hide. Drive a new alert insert; assert the count badge updates but the filtered list does not (since the new row is `active`).
- **Live region** — Vitest assertion that the live region's text content includes the alert summary on insert. Severity `critical` flips `aria-live` to `assertive`.

## Severity colour and motion

- Visual: severity badges and the bell dot use the three pinned tokens. Critical animates a subtle pulse (CSS keyframes, `prefers-reduced-motion: reduce` disables).
- Audible: a single short tone (e.g. 880 Hz, 250 ms with envelope) on critical insert. Reused via a single `AudioContext`. No looping.
- The severity-to-cue policy is hard-coded for V1; surface as a per-caregiver preference in V2 (see [BACKLOG.md](../../BACKLOG.md)).

## Risks

- **Notification permission UX.** Browsers throttle prompts; granted-once is sticky but denied-once is also sticky. Mitigation: prompt at the moment the first critical arrives (relevant context), not on app load. Add a one-line in-app banner explaining why the prompt appeared.
- **AudioContext autoplay policy.** Browsers require a user gesture to start an `AudioContext`. Mitigation: lazy-create the context on the first interaction during the session; if no interaction has happened yet when a critical arrives, fall back to the visual cue and live region. Caregivers actively monitoring will have interacted.
- **Cross-tab realtime race.** If tab A acks fast enough that the RPC returns before tab B's realtime channel sees the UPDATE, tab B's stale cache may briefly show the row as active. Mitigation: tab B's React Query cache invalidates on the realtime UPDATE event; if the realtime is delayed, the next refetch corrects. Test this explicitly.
- **Allocated-alerts subscription scaling.** One channel per allocated patient is fine for ≤ 5 patients. If demo scope grows, consolidate into a single channel filtered server-side via a `patient_id in (...)` filter (Supabase realtime supports it via `filter` strings). Defer until needed.
- **Brand red token clash.** If the existing palette already defines a red, reuse it rather than introducing a duplicate. Resolve in [CROSS_CUTTING.md §15](../CROSS_CUTTING.md#15-brand-consistency) review during PR.
- **Live region noise.** Polite-region announcements for every `info` alert can become noisy. Mitigation: throttle the live region to one announcement per 5 s for non-critical severities; criticals are not throttled.
- **`document.hidden` accuracy in iOS.** Mobile Safari handles `document.hidden` aggressively when the screen is off. Acceptable for V1; revisit if the prototype gets a mobile demo.
